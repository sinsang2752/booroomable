## 지금 작업 단계 (여기를 항상 먼저 확인)

> **현재: 2단계 — Supabase 멀티플레이 (마지막 항목만 남음)**
>
> 로비(방 생성/참가, Presence), 채팅(Broadcast), 상태 동기화(Postgres Changes), client_id 신원/재접속(로비 대기 중 + 게임 진행 중 모두), 자동 진행(턴 타이머)까지 완성했다. 남은 건 규칙 계산을 Edge Function(서버)으로 옮기는 것뿐 — 지금은 여전히 "지금 내 턴"인 클라이언트가 `src/game/engine.ts`를 직접 실행해서 결과를 DB에 쓰는 방식(클라이언트 신뢰 기반).

단계가 끝나면 이 섹션을 다음 단계로 갱신할 것. 전체 로드맵은 맨 아래 "개발 로드맵" 참고.

## 항상 지켜야 할 규칙

1. **한 번에 하나씩, 돌아가는 것부터.** 큰 기능을 통째로 만들지 말고, 작게 쪼개서 각 단계마다 실행되는 상태를 유지한다. 게임 개발 경험이 없는 개발자와 함께 작업 중임을 감안해 설명을 곁들인다.
2. **상태의 단일 출처(single source of truth).** 2단계 이후 게임의 진짜 상태는 항상 서버(Supabase DB)에 있고 클라이언트는 그것을 구독/표시할 뿐이다. 1단계에서는 로컬 상태로 시작하되, 나중에 서버로 옮기기 쉽게 게임 로직과 UI를 분리해서 짠다.
3. **밸런스 수치는 하드코딩하지 말고 한 곳에 모은다.** 초기 자본, 칸별 가격/통행료, 월급액 등은 설정 파일/상수 한 군데(예: `src/game/config.ts`, 보드 정의는 `src/game/board.ts`)에 모아 플레이하며 쉽게 조정할 수 있게 한다.
4. **큰 작업은 Plan 모드로 계획을 먼저 보여준다.**
5. 큰 변경 전에는 무엇을 왜 바꾸는지 한국어로 짧게 설명한다.

## 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 빌드/프론트 | Vite + React + TypeScript | 1단계부터 사용 |
| 백엔드/DB | Supabase (Postgres) | 2단계부터 |
| 실시간 로비 | Supabase Realtime Presence | 접속자 목록 |
| 실시간 채팅 | Supabase Realtime Broadcast | 방 채팅 |
| 게임 상태 동기화 | Supabase Postgres Changes | DB 변경 구독 |
| 부정행위 방지 | Supabase Edge Function | 2단계 이후 규칙 계산 |
| 데스크톱 포장 | Electron | 최종 단계 |

> 턴제라 밀리초 동기화가 불필요 → Socket.io는 쓰지 않고 Supabase Realtime으로 처리.

## 게임 규칙

### 보드
- **40칸 타원형(알약형) 트랙**: 위 19칸 + 아래 19칸 + 좌우 코너 각 1칸.
  - 좌 코너 = 출발, 우 코너 = 무인도.
- 데스크톱 하단 배치를 고려해 가로로 넓고 세로로 얇은 형태.
- 칸 종류(MVP 5종): `empty_land`(빈 땅) / 소유된 땅 / `start`(출발) / `jail`(무인도, 1턴 쉼) / (P1 이후: `tax`, `event`).

### 인원
- **최소 2명 ~ 최대 4명.** 2명부터 시작 가능. (정확히 4명이 차야 하는 게 아님)
- 말 색은 입장 순서대로 자동 배정: blue → red → green → amber.

### 이동 & 턴
- 주사위 2개. 더블(같은 눈)이면 한 번 더 굴림.
- 출발 칸을 지나가면 월급 지급.
- 주사위는 아직 클라이언트에서 생성 (지금 내 턴인 브라우저가 `Math.random`으로 굴림). Edge Function 이전 시 서버로 옮길 예정.

### 땅 & 통행료
- 빈 땅 도착 → 구매/패스 선택. 단, **땅값보다 잔액이 적으면 애초에 물어보지 않고 그냥 지나감** (살 수도 없는데 선택지를 보여줄 필요가 없어서, 2단계에서 추가).
- 남의 땅 도착 → 통행료 자동 징수.
- 통행료는 칸별 고정값으로 시작. (건물 업그레이드, 컬러그룹 독점 보너스는 P1)

### 자산 & 승패
- 초기 자본 균등 지급.
- 통행료를 못 내면 파산 → 탈락.
- 최후 1인 생존 시 승리. (P1: 턴 수 제한 후 자산 최고자 승리 옵션)

### 자동 진행 (방치형 핵심, 2단계에서 구현 완료)
- 각 턴 제한시간: 기본 30초, 방 설정으로 20~60초 (`rooms.turn_time_sec`, 로비에서 방장이 설정).
- 시간 초과 시 자동으로 주사위를 굴림 (구매 대기 중이었다면 자동 패스).
- **자리를 비워도 멈추지 않는 설계** (`src/hooks/useSyncedGame.ts`):
  1. 매 행동(굴리기/구매결정, 수동이든 자동이든) 성공 시 `rooms.turn_started_at`을 항상 `now()`로 갱신 — "이번 대기가 언제 시작됐는지"를 나타냄.
  2. 접속한 아무 클라이언트나 2초마다 "현재 시각 − turn_started_at > turn_time_sec?"을 확인.
  3. 초과 시 곧바로 자동 행동을 제출하지 않고, `rooms.version`을 조건부 업데이트(`WHERE version = 관찰한 값`)로 먼저 선점 — 여러 클라이언트가 동시에 감지해도 한 명만 성공 (Postgres가 같은 행에 대한 동시 UPDATE를 직렬화해줌). 처음 계획했던 `turn_number` 가드는 더블/구매대기처럼 턴 안에서 세부 단계가 바뀔 땐 값이 그대로라 세밀도가 부족해서 `version`으로 대체함.
  4. 선점에 성공한 클라이언트만 실제 자동 행동(`ROLL_DICE` 또는 `DECIDE_PURCHASE{buy:false}`)을 제출.

## 화면 & 레이아웃

### 흐름
```
[닉네임 입력] → [메인: 방 만들기 / 방 코드로 참가]
   → [로비] → 방장이 시작 → [게임] → [결과] → 메인 화면
```
(재접속: 새로고침해도 client_id로 아직 안 끝난 waiting/playing 방을 찾아 자동으로 로비/게임으로 복귀. "결과" 화면에서 새로고침하면 그 방은 이미 finished라 메인으로 떨어짐 — 의도된 단순화.)

### 로비 (우측 하단 세로 패널, ~320px)
방 코드(복사) / 참가자 목록(말 색·닉네임·준비상태·방장) / 턴 시간 설정(방장만) / 채팅 / 로비 나가기 / 시작 버튼(방장만).
- **방장은 준비 토글이 없음** — 시작 버튼만 갖고, 시작 조건은 "방장을 제외한 전원 준비완료".
- 로비 나가기 시 방장이 나가면 다음 순번에게 방장이 자동으로 넘어가고, 아무도 안 남으면 방이 삭제됨.

### 게임 (가로로 넓은 위젯)
- 타원 트랙이 중심. 가운데 공용 영역에 **현재 턴 정보(누구 차례·주사위·남은 시간) + 공용 채팅 입력창**.
  - "로그 열기" 버튼은 아직 없음 — 채팅은 저장 안 하고 Broadcast로만 오가서 다시 볼 로그 자체가 없음 (`chat_messages` 테이블 항목 참고).
- 각 플레이어 말 위:
  - **금액 배지: 항상 표시** (말 색과 같은 테두리, `balance` 실시간 반영).
  - **채팅 말풍선: 채팅 시 잠깐(3.5초) 떴다 사라짐.** 길이 제한 20자.
  - 파산자는 흐리게 + "파산" 표시.
- 좌우 코너(출발/무인도)는 아이콘 표시.
- 내 턴이 아니면 굴리기/구매 버튼 대신 "OO님 차례입니다" 문구만 보임 (다른 사람 턴에 조작 못 하게).

> 목업은 방향 확인용일 뿐. 색/간격/말풍선 스타일 등 세부는 구현하며 자유롭게 다듬어도 됨.

## 신원 관리 (로그인 없음)

- 로그인 계정 없음. 앱 첫 실행 시 랜덤 `client_id`를 생성해 localStorage에 저장.
- 이 `client_id`가 사람의 진짜 신원. 새로고침/재실행해도 이 ID로 게임 복귀.
- `nickname`은 표시용 이름.

## 데이터베이스 스키마 (2단계에서 Supabase에 생성)

> 성격 구분: **신원**(사람) / **게임상태**(이번 판 한정) / **외형**(표시용 꾸미기).

### `rooms`
`id` uuid PK · `code` text · `status` text(waiting/playing/finished) · `turn_time_sec` int · `current_player_id` uuid · `turn_number` int · `turn_started_at` timestamp · `host_client_id` text · `created_at` timestamp

2단계 상태 동기화/자동진행을 위해 추가된 컬럼(엔진의 `GameState`를 그대로 담기 위함, `src/game-sync/mapping.ts` 참고):
`phase` text(awaiting-roll/awaiting-purchase-decision/game-over) · `last_roll_d1` int · `last_roll_d2` int · `is_double_roll` bool · `pending_purchase_tile_idx` int · `winner_player_id` uuid · `notice` text · `version` int (낙관적 동시성 가드, 액션마다 1씩 증가)

### `players`
| 컬럼 | 타입 | 성격 |
|---|---|---|
| id | uuid PK | |
| room_id | uuid FK→rooms | |
| client_id | text | 신원 (재접속 복구) |
| nickname | text | 신원 |
| appearance | jsonb | 외형 `{hat, face, body, ...}` (커스텀 확장용) |
| color | text | 게임상태 (자동배정: blue/red/green/amber) |
| position | int | 게임상태 (0~39) |
| balance | int | 게임상태 |
| is_bankrupt | bool | 게임상태 |
| skip_next_turn | bool | 게임상태 (무인도 착지 시 다음 턴 스킵) |
| seat_order | int | 게임상태 |
| is_ready | bool | 게임상태 (로비 준비완료, 방장은 해당 없음) |
| is_connected | bool | 게임상태 (컬럼만 있고 아직 실제로 갱신하는 코드는 없음 — 로비는 Presence로 접속 상태를 별도 판단) |
| created_at | timestamp | |

- `appearance`를 jsonb로 두는 이유: 커스텀 항목이 늘어도 DB 스키마 변경 불필요.
- `color`를 appearance 밖에 두는 이유: 통행료·말 렌더링에서 자주 쓰는 게임 로직상의 값.
- `unique(room_id, seat_order)`, `unique(room_id, client_id)` 제약으로 동시 참가 시 좌석 충돌/중복 참가 방지.
- 확장 메모: 나중에 로그인 + 캐릭터 재사용 단계에서 신원+외형을 별도 `profiles` 테이블로 분리하고 `players`는 게임상태만 남기도록 리팩터링 예정.

### `tiles` (40칸 보드 정의, 고정 마스터 데이터)
`idx` int PK(0~39) · `name` text · `type` text · `price` int · `toll` int · `color_group` text
> DB에 생성 및 시드는 끝났지만(`src/game/board.ts`·`config.ts` 공식과 동일한 값), **클라이언트는 아직 이 테이블을 안 읽는다** — `engine.ts`가 여전히 로컬 상수 `BOARD`(`src/game/board.ts`)를 직접 참조. 지금은 둘이 같은 공식으로 만들어져 값이 같지만, 나중에 클라이언트가 DB에서 직접 읽도록 바꾸는 게 완전한 마이그레이션.

### `ownerships`
`id` uuid PK · `room_id` uuid FK · `tile_idx` int FK · `player_id` uuid FK · `level` int(프로토타입 0, 건물은 P1)
`unique(room_id, tile_idx)` 제약으로 같은 칸 중복 구매 방지.

### `chat_messages` (선택)
`id` uuid PK · `room_id` uuid FK · `player_id` uuid FK · `body` text · `created_at` timestamp
> 실시간 채팅은 Broadcast로 즉시 전파가 기본. 이 테이블은 로그 다시보기 저장용. 초기엔 생략 가능.
> 2단계 채팅 슬라이스에서 실제로 생략함 — 현재 채팅은 Broadcast(`room:{roomId}:chat` 채널)로만 오가고 이 테이블엔 저장하지 않는다. 그래서 로비→게임 전환/새로고침 시 이전 대화가 사라진다. 나중에 로그 다시보기가 필요해지면 메시지 전송 시 이 테이블에 insert 한 줄 추가 + "로그 열기" UI만 얹으면 된다.

## 보안 (단계적 적용)

- 부정행위 방지: 주사위·구매·통행료 계산을 클라이언트가 하면 조작 가능.
  - 1단계: 클라이언트 처리 허용.
  - 2단계: 규칙 계산을 Edge Function(서버)으로 이전.
- RLS(행 수준 접근 제어): 프로토타입은 느슨하게 시작 → 이후 강화.

## 개발 로드맵

- **1단계 (완료)** — 로컬 패스앤플레이. 40칸 보드, 주사위/이동/구매/통행료/파산/승리. 한 화면에서 턴 넘기며 규칙 검증.
- **2단계 (진행 중, 마지막 항목만 남음)** — Supabase 멀티플레이: ~~로비(방 생성/참가, Presence)~~ ~~채팅(Broadcast)~~ ~~상태 동기화(Postgres Changes)~~ ~~client_id 신원/재접속~~ ~~자동 진행(턴 타이머)~~ 전부 완료, **규칙 계산 Edge Function 이전만 남음**.
- **3단계 (P1)** — 재미 요소: 황금열쇠, 건물 업그레이드, 컬러그룹 독점 보너스, 세금.
- **4단계 (P2)** — Electron 포장(투명·항상위·작은 창, 트레이, 내 턴 알림), 캐릭터 외형 커스텀 UI(`appearance` 활용).

## 미정 사항 (구현하며 결정)

- 밸런스 수치 — 1단계에서 초기값 확정: 초기자본 1500 / 월급 200 / 땅값은 100부터 20씩 증가 / 통행료는 땅값의 20% (`src/game/config.ts`). 플레이 시뮬레이션 결과 2인전에서는 시간이 지날수록 월급 누적으로 파산이 잘 안 나 게임이 길어지는 경향 확인 — 통행료 비율 상향 등 추가 튜닝 필요.
- 승리 조건 — 1단계에서 "최후 1인 생존"으로 확정 구현. 턴 수 제한 후 자산 최고자 방식은 P1 이후 옵션으로 남겨둠.
- 로비→게임 창 전환 방식(Electron 단계): 같은 창 리사이즈 vs 별도 창. (2단계 이후 결정)
