## 지금 작업 단계 (여기를 항상 먼저 확인)

> **현재: 2단계 완료 → 3단계(P1) 진행 중**
>
> 로비, 채팅, 상태 동기화, client_id 신원/재접속, 자동 진행(턴 타이머), 규칙 계산 Edge Function 이전까지 2단계 전 항목 완료. 규칙 검증(누구 턴인지, 방장인지 등)과 실제 DB 쓰기는 이제 클라이언트가 아니라 `supabase/functions/game-action`(서비스 롤 키)이 전담하고, RLS+컬럼 권한으로 게임 상태 컬럼은 anon 키로 직접 못 쓰게 잠갔다.

**3단계 구현 순서 (한 슬라이스씩, 각 슬라이스마다 플레이 가능한 상태 유지)**

1. **건물 시스템** ← 여기부터. 나머지 룰이 전부 건물 수/등급에 의존하므로 반드시 먼저.
   - **구현 완료**: 자기 땅 재방문 시 한 등급씩 업그레이드(레벨 0~4, 통행료/건설비 배율은 아래 표 그대로) + **최초 구매 시 감당되는 등급 중 목표를 한 번에 선택해 즉시건축**(`awaiting-initial-build-decision` phase) + **출발칸 정확 도착 보너스**(`awaiting-start-bonus-build` phase, 보드에서 업그레이드할 땅을 직접 클릭해서 고름). 세 흐름이 공유하는 "업그레이드 가능한가/비용" 계산은 `src/game/buildings.ts`에 순수 함수로 모아둠.
   - 파산 방지용 건물/땅 자동매각도 구현 완료 (아래 "매각 & 파산" 참고).
   - `landmark` 칸(선택 도입 항목)은 **일단 건너뛰기로 결정** — 도입 여부/위치는 여전히 미정으로 남겨둠.
2. **황금열쇠** — **1차 + 2차 전부 구현 완료** (아래 "황금열쇠" 참고). 3단계 로드맵 항목 중 완전히 끝난 두 번째 항목.
3. **사회복지기금** — **구현 완료** (아래 "사회복지기금" 참고).
4. **우주여행** — **구현 완료** (아래 "우주여행" 참고).
5. **더블 3연속 → 무인도**, 무인도 규칙 확장 — **구현 완료** (아래 "무인도" 참고).
6. **독점 보너스 / 승리 조건 추가** (아래 "독점 보너스" 참고 — 채택 여부 미정) ← 다음은 여기부터. 3단계 로드맵의 마지막 남은 항목.

**이번 단계에서 구현하지 않기로 확정한 것** (원작에 있지만 방치형과 안 맞아 제외):
자기 턴에 아무 땅에나 상시 건물 짓기(단 **출발칸 정확 도착 시 1등급 보너스는 예외로 도입** — "건물 시스템" 참고) / 우대권 / 무인도 탈출권(무전기) / 대출·담보대출.

**자동 기본값의 대원칙**: 타임아웃으로 자동진행되면 **아무것도 하지 않고 턴을 넘긴다.** 구매·건축·우주여행 이동 전부 동일. 방치한 쪽이 이득을 보는 상황이 생기지 않도록 예외를 두지 않는다.

단계가 끝나면 이 섹션을 다음 단계로 갱신할 것. 전체 로드맵은 맨 아래 "개발 로드맵" 참고.

## 항상 지켜야 할 규칙

1. **한 번에 하나씩, 돌아가는 것부터.** 큰 기능을 통째로 만들지 말고, 작게 쪼개서 각 단계마다 실행되는 상태를 유지한다. 게임 개발 경험이 없는 개발자와 함께 작업 중임을 감안해 설명을 곁들인다.
2. **상태의 단일 출처(single source of truth).** 2단계 이후 게임의 진짜 상태는 항상 서버(Supabase DB)에 있고 클라이언트는 그것을 구독/표시할 뿐이다. 1단계에서는 로컬 상태로 시작하되, 나중에 서버로 옮기기 쉽게 게임 로직과 UI를 분리해서 짠다.
3. **밸런스 수치는 하드코딩하지 말고 한 곳에 모은다.** 초기 자본, 칸별 가격/통행료, 월급액 등은 설정 파일/상수 한 군데(예: `src/game/config.ts`, 보드 정의는 `src/game/board.ts`)에 모아 플레이하며 쉽게 조정할 수 있게 한다.
4. **모든 선택지에는 "자동 기본값"이 반드시 정의되어야 한다.** 이 게임은 방치형이라 턴 제한시간이 지나면 시스템이 대신 결정한다. 새 룰을 추가할 때 플레이어의 판단이 필요한 지점이 생기면, 그 지점의 자동 동작을 반드시 같이 정의할 것. 자동 기본값을 정할 수 없을 만큼 복잡한 선택이라면 그 룰 자체가 이 게임에 안 맞는다고 보고 단순화한다. (원작 부루마불 룰 중 상당수가 여기에 걸려서 각색이 필요함)
5. **큰 작업은 Plan 모드로 계획을 먼저 보여준다.**
6. 큰 변경 전에는 무엇을 왜 바꾸는지 한국어로 짧게 설명한다.

## 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 빌드/프론트 | Vite + React + TypeScript | 1단계부터 사용 |
| 백엔드/DB | Supabase (Postgres) | 2단계부터 |
| 실시간 로비 | Supabase Realtime Presence | 접속자 목록 |
| 실시간 채팅 | Supabase Realtime Broadcast | 방 채팅 |
| 게임 상태 동기화 | Supabase Postgres Changes | DB 변경 구독 |
| 부정행위 방지 | Supabase Edge Function | 완료 — `supabase/functions/game-action` |
| 데스크톱 포장 | Electron | 최종 단계 |

> 턴제라 밀리초 동기화가 불필요 → Socket.io는 쓰지 않고 Supabase Realtime으로 처리.

## 게임 규칙

### 보드
- **40칸 타원형(알약형) 트랙**: 위 19칸 + 아래 19칸 + 좌우 코너 각 1칸.
  - 좌 코너 = 출발, 우 코너 = 무인도.
- 데스크톱 하단 배치를 고려해 가로로 넓고 세로로 얇은 형태.
- 칸 종류
  - 2단계까지: `empty_land`(빈 땅) / 소유된 땅 / `start`(출발) / `jail`(무인도).
  - 3단계 추가: `event`(황금열쇠, **구현 완료** — `EVENT_TILE_INDICES`) / `welfare_pay`·`welfare_get`(사회복지기금, **구현 완료** — `WELFARE_PAY_TILE_IDX`/`WELFARE_GET_TILE_IDX`) / `space_travel`(우주여행, **구현 완료** — `SPACE_TRAVEL_TILE_IDX`) / `landmark`(건물 못 짓는 고가 특수 땅, 도입 보류).
- **특수칸 배치 주의**: 원작은 정사각형이라 네 모서리에 출발·무인도·기금수령·우주여행을 배치하지만, 우리 보드는 코너가 좌우 2개(출발·무인도)뿐이다. 나머지 특수칸은 위/아래 줄 안에 배치해야 하며, 한쪽에 몰리지 않게 **황금열쇠 칸은 트랙 전체에 고르게 분산**시킬 것(원작도 각 구역마다 배치되어 있음).

### 인원
- **최소 2명 ~ 최대 4명.** 2명부터 시작 가능. (정확히 4명이 차야 하는 게 아님)
- 말 색은 입장 순서대로 자동 배정: blue → red → green → amber.

### 이동 & 턴
- 주사위 2개. 더블(같은 눈)이면 한 번 더 굴림.
- 출발 칸을 지나가면 월급 지급. **출발 칸에 정확히 멈추면** 월급에 더해 자유 건축 보너스가 발동한다("건물 시스템" 참고) — 지나가는 것과 정확히 멈추는 것을 구분해서 처리할 것.
- 주사위는 이제 서버(`game-action` Edge Function)에서 생성. 브라우저는 "굴려줘" 요청만 보내고, 실제 `Math.random` 호출과 결과 계산은 Deno 서버 안에서 일어난다.

### 땅 & 통행료
- 빈 땅 도착 → 구매/패스 선택. 단, **땅값보다 잔액이 적으면 애초에 물어보지 않고 그냥 지나감** (살 수도 없는데 선택지를 보여줄 필요가 없어서, 2단계에서 추가).
- 남의 땅 도착 → 통행료 자동 징수.
- 통행료는 소유 등급(건물)에 따라 결정. 아래 "건물 시스템" 참고.

### 건물 시스템 (3단계, 원작 부루마불 기준으로 각색)

**등급**: `level` 0~4 — 0=땅만(대지료) / 1=별장1 / 2=별장2 / 3=빌딩 / 4=호텔.
- 원작대로 한 땅에 건물은 한 채까지이며, 별장만 예외적으로 2채가 가능하다. 그래서 별장1·별장2를 별도 등급으로 둔다.
- 상위 등급을 지으면 하위 건물은 대체된다(철거·재건축 정산 없음 — 단순화).

**언제 지을 수 있나 — 도착 시에만 (확정. 원작 정식룰과 다름, 의도된 각색)**
- 원작 정식룰은 자기 턴이면 말 위치와 무관하게 자기 땅 아무 곳에나 건물을 지을 수 있고, 나무위키는 "도착한 땅에만 짓기"를 게임을 지루하게 만든 변형룰이라고 비판한다. 하지만 이 프로젝트는 **방치형**이라 "내 땅 여러 곳 중 어디에 무엇을 지을지"를 매 턴 고르게 만들 수 없고, 자동 기본값도 정하기 어렵다. → **자기 소유 땅에 도착했을 때만 그 땅을 업그레이드**하는 방식을 채택한다. **매 턴 자유 건축은 채용하지 않는다 — 확정.** (단 출발칸 정확 도착 시에는 예외적으로 자유 건축 1회 허용. 아래 참고)
- 대신 원작 변형룰의 "게임이 늘어진다"는 부작용은 아래 두 가지로 상쇄한다:
  - **최초 구매 시 자금이 허용하는 범위에서 건물까지 한 번에 지을 수 있다** (모바일 모두의마블의 '신의 손' 계열 룰). 땅만 사고 4번을 더 방문해야 호텔이 되는 구조보다 훨씬 빠르다. **UI는 한 단계씩 반복해서 묻지 않고, 감당 가능한 등급(별장1/별장2/빌딩/호텔) 중 원하는 목표를 한 번에 선택하는 방식**(`awaiting-initial-build-decision` phase, 액션 `DECIDE_INITIAL_BUILD{targetLevel}`) — 서버가 `targetLevel`을 그대로 안 믿고 감당 가능한 최고 레벨로 다시 클램프한다.
  - 재방문 시에는 **한 번에 한 등급씩** 상승(Y/N만 묻는 기존 방식 그대로, `awaiting-build-decision`).
- **자동 기본값**: 자동진행으로 넘어간 경우 건물은 짓지 않는다(패스). 구매도 현행대로 자동 패스.

**예외 — 출발칸 정확 도착 보너스 (구현 완료)**
주사위 눈이 딱 맞아 **출발 칸에 정확히 멈춘 경우에 한해, 자기 소유 땅 중 아무 곳이나 하나를 골라 한 등급 올릴 수 있다.** (모바일 모두의마블의 출발지 보너스 규칙. 원작 부루마불에도 특수능력 변형룰로 소개되어 있음)
- 상시 자유 건축은 방치형과 안 맞아 채용하지 않지만, **출발칸 정확 도착이라는 낮은 빈도로 제한하면** 부담 없이 전략성만 챙길 수 있다. 한 바퀴에 한 번 올까 말까 한 기회라 "이번엔 어디에 지을까"를 고민할 만한 순간이 된다.
- 세부 규칙:
  - **정확히 멈춰야** 한다. 지나가기만 한 경우는 해당 없음(월급은 지나가도 받지만 이 보너스는 별개).
  - **한 등급만** 상승. 여러 채를 한 번에 짓거나 여러 땅에 나눠 짓는 것은 불가.
  - 건설비는 정상 지불. 자금이 부족하면 그 등급은 선택할 수 없다.
  - 대상에서 제외: 이미 호텔(level 4)인 땅, `landmark`(건물 못 짓는 땅), 소유하지 않은 땅.
  - 지을 수 있는 땅이 하나도 없으면 선택 단계를 건너뛴다.
- **자동 기본값**: 제한시간 내에 고르지 않으면 **아무것도 짓지 않고 그대로 턴을 넘긴다.**
- UI: 보드에서 업그레이드 가능한 자기 땅을 하이라이트(테두리 펄스 효과)하고, 칸에 마우스를 올리면 다음 등급/건설비가 툴팁으로 표시됨. "건너뛰기" 버튼도 제공.
- `landmark` 칸은 아직 미도입이라 현재 "제외 대상"은 실질적으로 이미 호텔인 땅/소유하지 않은 땅 두 가지만 적용됨.

**통행료 배율 (원작 실측 비율 기반, `config.ts`에 상수로)**
원작 수치를 보면 건물 없는 대지료는 땅값의 6~10% 수준으로 매우 낮고, 호텔은 땅값의 4~5배까지 뛴다(예: 런던 땅값 35만 → 대지료 3.5만, 호텔 통행료 150만). 이 급격한 상승 곡선이 파산을 만드는 장치다. 우리 공식은 다음 배율로 시작하고 플레이하며 조정한다:

| level | 건물 | 통행료 (땅값 대비) | 건설비 (땅값 대비) |
|---|---|---|---|
| 0 | 없음 | ×0.1 | — |
| 1 | 별장1 | ×0.5 | ×0.4 |
| 2 | 별장2 | ×1.5 | ×0.4 (추가) |
| 3 | 빌딩 | ×3 | ×1.5 |
| 4 | 호텔 | ×4.5 | ×2.5 |

> **중요**: 현재 `config.ts`의 "통행료 = 땅값의 20% 고정"은 이 표의 level 0(×0.1)과 상충한다. 건물 시스템을 넣으면서 level 0 통행료를 낮추고 상한을 열어야 "월급 누적으로 파산이 안 나 게임이 길어지는" 기존 문제가 해소된다. 밸런스 튜닝의 핵심 지점.

**건물을 지을 수 없는 특수 땅 (선택 도입)**
원작에는 건물 건설이 불가능한 대신 땅값·통행료 자체가 높은 땅들이 있다(대표적으로 서울: 땅값 100만·통행료 200만으로 통행료가 땅값의 2배). 게임의 상징이지만 밸런스 파괴로도 유명하다.
- 도입한다면 `tiles.type = 'landmark'`로 1~2칸만. 통행료는 땅값의 1.5~2배.
- 강력한 만큼 위치가 중요하다 — 출발 칸 바로 앞에 두면 한 바퀴마다 위협이 되어 게임이 빨리 끝난다.

### 매각 & 파산 (구현 완료)
- 통행료를 낼 현금이 부족하면 **자동으로 낮은 등급 건물부터 매각**해 충당한다. 매각가는 건설비의 100%(`BUILDING_SALE_RATIO`, `config.ts`. 모노폴리처럼 반액으로 할지는 플레이해보고 밸런스 보고 결정).
- 건물을 다 팔아도 부족하면 **땅 매각**(`LAND_SALE_RATIO`, 땅값의 100%), 그래도 부족하면 파산.
- **자동 기본값**: 위 순서(낮은 등급 건물 → 높은 등급 건물 → 싼 땅 → 비싼 땅)로 시스템이 알아서 처분. 플레이어에게 무엇을 팔지 묻지 않는다(방치형이라). `src/game/buildings.ts`의 `liquidateForDebt` 순수 함수 + `engine.ts`의 통행료 분기에서 호출.
- 구현하면서 발견한 기존 버그: `computePatches`(`src/game-sync/mapping.ts`)가 원래 "액션 한 번에 칸 하나만 바뀐다"고 가정해 변경된 칸을 하나만 찾고 멈추는 구조였는데, 매각은 한 번에 여러 칸(건물 여러 채 + 땅 여러 곳)이 동시에 바뀔 수 있어 이번에 배열(`ownershipUpserts`/`ownershipDeletions`)로 고쳤다. 이 컬럼들이 없었으면 매각 결과 일부가 DB에 조용히 반영 안 될 뻔했음.
- **대출 시스템은 구현하지 않는다 (확정).** 원작에는 은행 대출(1회 한정, 3턴 내 상환) 룰이 있지만, 상환 판단의 자동 기본값을 정하기 어렵고 게임이 늘어진다. 담보대출 등 변형룰도 마찬가지로 전부 제외.

### 황금열쇠 (3단계)

`tiles.type = 'event'` 칸(`config.ts`의 `EVENT_TILE_INDICES = [5, 15, 25, 35]`, 트랙에 고르게 4곳)에 도착하면 카드 1장을 뽑아 즉시 효과 적용. 덱은 셔플 후 순환.

**1차 구현 — 전부 완료**
- **상금(구현 완료)**: 고정액 수령. 원작 예시(노벨상 30만/복권 20만/장학금 10만/연금 5만)를 우리 경제 규모(초기자본 1500)에 맞춰 1/1000 스케일 → 노벨상 300 / 복권 200 / 장학금 100 / 연금 50 (`config.ts`의 `GOLDEN_KEY_DECK`).
- **벌금(구현 완료)**: 고정액 지불(해외유학 100 / 병원비 50, 같은 스케일). 통행료와 똑같은 "빚"으로 취급해 잔액이 부족하면 건물 시스템의 자동매각(`liquidateForDebt`)이 먼저 발동하고, 그래도 부족하면 파산.
- **생일축하(구현 완료)**: 생존한 다른 모든 플레이어에게서 30씩 징수. 각자 가진 만큼만 내고(부족해도 파산까지는 안 감 — 문서에 부족분 규정이 없어 단순화) 현재 플레이어에게 합산 지급.
- **이동(구현 완료)**: 출발지로 이동(월급 받음 + 출발칸 정확 도착 보너스 체크까지 이어짐) / 무인도로 이동(월급 없음, 다음 턴 스킵) / 뒤로 2칸 · 뒤로 3칸 이동(월급 없음, 별개 카드 2장으로 "2~3칸"의 다양성을 자연스럽게 구현).
  - 원작 규칙 그대로: 이동 카드는 "목적지에 닿을 때까지 계속 전진"하는 개념이라, **이동한 칸에 도착하면 그 칸의 정상 효과(통행료 지불/구매 프롬프트/건물 업그레이드 프롬프트 등)까지 이어서 처리**한다.
  - 구현을 위해 `handleRollDice`에 있던 "칸 도착 효과" 로직을 `resolveTileLanding` 함수로 뽑아 리팩터링(`src/game/engine.ts`) — 주사위 이동과 황금열쇠 이동 카드 양쪽에서 재사용. 황금열쇠 이동 카드가 하필 다른 황금열쇠 칸에 떨어지는 극히 드문 경우엔 카드를 또 뽑지 않고 지나간다(재귀 방지, 문서에 규정 없는 단순화).
  - **`특정 칸으로 이동` 카드는 도입하지 않음(확정)** — 원작의 철도·전기회사처럼 우리 보드에 없는 "지정 대상"을 가리켜야 하는 카드라 뭘 가리킬지 불명확해서 스킵.
- 카드 덱은 `rooms.event_deck`(jsonb, 3단계에서 신규 추가된 컬럼)에 안 뽑은 카드 순서를 서버가 들고 있다가 셔플 후 순환(`src/game/engine.ts`의 `shuffleDeck`, 주사위와 동일하게 서버에서만 `Math.random()` 실행).

**2차 구현 — 전부 완료**
- **건물유지비 3종(구현 완료)**: 보유 건물 등급(별장=레벨1·2 공통/빌딩=레벨3/호텔=레벨4)별 단가 × 개수를 합산해 은행에 지불. 원작 비율을 1/1000 스케일 → 정기종합소득세(별장30/빌딩100/호텔150) > 건물수리비(30/60/100) > 방범비(10/30/50). 벌금과 동일하게 "빚"으로 취급해 부족하면 자동매각(`liquidateForDebt`) 후 그래도 부족하면 파산.
- **반액대매출(구현 완료)**: 보유한 땅 중 총 가치(땅값+누적 건설비, `src/game/buildings.ts`의 `getPropertyValue`)가 가장 큰 곳을 골라 그 절반을 현금으로 받고 소유권·건물을 통째로 잃는다(다시 "빈 땅"으로 돌아가 누구나 살 수 있음). 소유한 땅이 없으면 아무 일도 안 일어남.

**미도입 (확정)**
- **우대권**(통행료 1회 면제), **무인도 탈출권(무전기)** 은 구현하지 않는다. 보유했다가 원하는 때 쓰는 카드라 "언제 쓸지"가 방치형과 충돌하고, 보유 아이템 상태를 따로 관리해야 해서 복잡도만 늘어난다. → 황금열쇠 덱에서 제외.

### 우주여행 (구현 완료)

`type = 'space_travel'` 칸(idx18, `SPACE_TRAVEL_TILE_IDX`)에 도착하면 **보드의 원하는 칸으로 이동**한다. 원작에서 가장 전략적인 칸.

**진행 방식**
- 도착 즉시 이동지를 정한다(새 phase `awaiting-space-travel-destination`, 액션 `DECIDE_SPACE_TRAVEL{tileIdx}`).
- **우주여행 칸으로는 다시 갈 수 없다** — 목적지 후보에서 아예 제외(서버가 `tileIdx`로 우주여행 칸이나 현재 위치를 보내면 무시하고 "이동 안 함"과 동일하게 처리, 클라이언트를 신뢰하지 않음).
- 이동은 텔레포트가 아니라 **트랙을 따라 전진하는 판정**이다. `목적지 idx < 현재 위치`면 앞으로 걷다가 39→0으로 넘어간 것이므로 월급을 받는다(원작 규칙과 동일).
- 도착한 칸의 효과는 정상 적용 — 통행료/구매 프롬프트/건물 프롬프트/출발점 보너스/황금열쇠까지 전부 `resolveTileLanding` 공용 함수로 자연스럽게 이어짐(칸 도착 효과 로직을 함수로 뽑아둔 덕에 새 코드 없이 재사용).
- **이동 후에는 더블이어도 추가 굴림 없이 턴을 종료한다** — `isDoubleRoll`을 이동 시점에 강제로 `false`로 덮어써서, 도착한 칸이 또 구매/건축 프롬프트로 이어지는 경우까지 전부 재굴림 없이 처리되도록 보장(체인 도중 어디서 최종 `finishTurnStep`이 불려도 안전).
- 원작의 "컬럼비아호 소유자에게 이용료 지불"은 탈것 개념이 없으므로 **생략**.

**자동 기본값 (방치형 핵심)**
- 제한시간 내에 목적지를 고르지 않으면 **아무것도 하지 않고 그대로 턴을 넘긴다.**

**UI**
- 출발점 보너스 때 만든 보드 클릭 선택 메커니즘(`selectableTiles`/`onSelectTile`)을 그대로 재사용 — 목적지 후보 전부(우주여행 칸·현재 위치 제외) 하이라이트, 잔액으로 못 사는 빈 땅도 선택 가능(이동 자체는 안 막음).
- "이동 안 함" 버튼 제공.

### 사회복지기금 (구현 완료, 기존 "세금" 항목을 대체)
원작의 기부 칸 + 접수 칸 구조. 세금을 그냥 은행에 버리는 것보다 훨씬 재밌고 구현도 단순하다.
- `type = 'welfare_pay'` 칸(idx10): 도착 시 고정액(`WELFARE_PAY_AMOUNT`=100)을 **기금 풀에 적립**(`rooms.welfare_pool`). 통행료·황금열쇠 벌금과 동일하게 "빚"으로 취급해 잔액 부족하면 자동매각(`liquidateForDebt`) 후에도 부족하면 파산.
- `type = 'welfare_get'` 칸(idx30): 도착 시 **적립된 기금 전액 수령**. 비어 있으면 아무 일도 없음.
- 황금열쇠 덱에 "사회복지기금 접수처로 이동" 카드 추가 — 다른 이동 카드처럼 도착한 칸(항상 `welfare_get`)의 효과가 이어지고, 이동 중 출발점을 지나치면 월급도 받는다(일반 이동 카드 규칙과 동일, 무인도행만 예외).

### 무인도 (구현 완료)
- 원작 그대로 **3턴 대기**(`players.jail_turns_left`, 무인도 도착 시 3으로 설정). 예전 1턴 스킵용 `skip_next_turn` bool은 폐기 예정이었으나 컬럼 자체는 안전하게 남겨두고(DROP 안 함) 코드에서만 안 쓴다.
- **핵심 변경**: 무인도에 갇힌 플레이어도 이제 자기 턴마다 실제로 주사위를 굴린다(기존엔 턴 자체가 로테이션에서 통째로 건너뛰어져 굴릴 기회가 없었음). 더블이면 그 눈만큼 이동해 즉시 탈출, 아니면 이동 없이 대기 턴만 1 줄고, 3번째 시도에서도 더블이 안 나오면 그 굴림으로 강제 석방+이동한다(원작과 동일 — 대출이 없는 이 프로젝트 특성상 "3번째엔 벌금 내고 나가기" 대신 그냥 무료로 풀어줌). 탈출/석방 이동은 `resolveTileLanding`을 그대로 재사용해 도착 칸 효과(구매/건축/황금열쇠 등)까지 정상 적용하되, **보너스 재굴림은 절대 없음**(탈출에 쓴 더블이든 이후 정상 착지가 우연히 또 더블이었든 전부 `isDoubleRoll`을 강제로 꺼서 처리 — 안 그러면 방금 들어간 무인도를 같은 턴에 또 탈출 시도하는 버그가 생김. 구현 중 실제로 이 버그를 잡아서 고쳤음).
- 새 phase/액션 없이 기존 `ROLL_DICE` + `awaiting-roll`만으로 전부 표현됨 — 타임아웃 자동진행도 무변경으로 그대로 작동(무인도 중이어도 시간 초과 시 서버가 대신 `ROLL_DICE`를 제출).
- 원작에서 무인도는 후반부에 오히려 피난처가 된다(건물이 깔린 판을 안전하게 넘길 수 있어서). 이 성격은 그대로 살아있다.
- **더블 3연속 → 무인도 강제 이동 (구현 완료)** — `rooms.consecutive_doubles`로 추적, 턴이 실제로 다음 사람에게 넘어갈 때 0으로 리셋. 3번째 연속 더블이면 이동 계산 자체를 안 하고 바로 무인도 칸으로 텔레포트(`resolveTileLanding` 호출 안 함) — 그래서 "출발지를 지나쳐도 월급을 받지 못한다"는 요건이 코드 구조상 자동으로 충족됨.
- UI: 말 배지에 "무인도 N" 표시 추가(`PlayerToken.tsx`, 파산 배지와 같은 자리, 탈락자와 헷갈리지 않게 다른 색). 그 외 별도 프롬프트 없음 — notice 텍스트로 상황 설명.

### 독점 보너스 / 추가 승리 조건 (채택 여부 미정)
- 기존 로드맵의 "컬러그룹 독점 보너스(같은 색 전부 소유 시 통행료 2배)"는 **원작 부루마불에는 없는 모노폴리 룰**이다. 건물 시스템이 들어오면 통행료가 이미 땅값의 4.5배까지 오르므로, 여기에 독점 2배까지 겹치면 과할 수 있다. → 건물 시스템 붙인 뒤 플레이해 보고 결정.
- 대안: 모두의마블식 **독점 즉시 승리** — 같은 컬러그룹을 전부 소유하면 그 자리에서 승리. 방치형 게임에 명확한 종료 조건을 하나 더 주므로 "게임이 길어진다" 문제에 오히려 잘 맞는다.
- 둘 다 넣지 않고 건물 시스템만으로 밸런스가 잡히면 그대로 두는 것도 정답.

### 자산 & 승패
- 초기 자본 균등 지급.
- 통행료를 못 내면 파산 → 탈락.
- **게임 도중 언제든 "게임 포기하기"로 자진 탈락 가능** (2단계에서 추가). 확인창 한 번 거친 뒤 파산과 동일하게 처리(`isBankrupt=true`, 잔액 0). 내 턴에 포기하면 턴이 다음 사람에게 넘어가고, 남의 턴에 포기해도 진행 중인 턴은 안 끊김.
- 최후 1인 생존 시 승리. (P1: 턴 수 제한 후 자산 최고자 승리 옵션)

### 자동 진행 (방치형 핵심, 2단계에서 구현 완료)
- 각 턴 제한시간: 기본 30초, 방 설정으로 20~60초 (`rooms.turn_time_sec`, 로비에서 방장이 설정).
- 시간 초과 시 자동으로 주사위를 굴림 (구매 대기 중이었다면 자동 패스).
- **자리를 비워도 멈추지 않는 설계** (`src/hooks/useSyncedGame.ts`):
  1. 매 행동(굴리기/구매결정, 수동이든 자동이든) 성공 시 `rooms.turn_started_at`을 항상 `now()`로 갱신 — "이번 대기가 언제 시작됐는지"를 나타냄.
  2. 접속한 아무 클라이언트나 2초마다 "현재 시각 − turn_started_at > turn_time_sec?"을 확인.
  3. 초과 시 곧바로 자동 행동을 제출하지 않고, `rooms.version`을 조건부 업데이트(`WHERE version = 관찰한 값`)로 먼저 선점 — 여러 클라이언트가 동시에 감지해도 한 명만 성공 (Postgres가 같은 행에 대한 동시 UPDATE를 직렬화해줌). 처음 계획했던 `turn_number` 가드는 더블/구매대기처럼 턴 안에서 세부 단계가 바뀔 땐 값이 그대로라 세밀도가 부족해서 `version`으로 대체함.
  4. 선점에 성공한 클라이언트만 실제 자동 행동(`ROLL_DICE` 또는 `DECIDE_PURCHASE{buy:false}`)을 제출.

## 화면 & 레이아웃

### 흐름
```
[닉네임 입력] → [메인: 방 만들기 / 방 코드로 참가]
   → [로비] → 방장이 시작 → [게임] → [결과] → 메인 화면
```
(재접속: 새로고침해도 client_id로 아직 안 끝난 waiting/playing 방을 찾아 자동으로 로비/게임으로 복귀. "결과" 화면에서 새로고침하면 그 방은 이미 finished라 메인으로 떨어짐 — 의도된 단순화.)

### 로비 (우측 하단 세로 패널, ~320px)
방 코드(복사) / 참가자 목록(말 색·닉네임·준비상태·방장) / 턴 시간 설정(방장만) / 채팅 / 로비 나가기 / 시작 버튼(방장만).
- **방장은 준비 토글이 없음** — 시작 버튼만 갖고, 시작 조건은 "방장을 제외한 전원 준비완료".
- 로비 나가기 시 방장이 나가면 다음 순번에게 방장이 자동으로 넘어가고, 아무도 안 남으면 방이 삭제됨.

### 게임 (가로로 넓은 위젯)
- 타원 트랙이 중심. 가운데 공용 영역에 **현재 턴 정보(누구 차례·주사위·남은 시간) + 공용 채팅 입력창**.
  - "로그 열기" 버튼은 아직 없음 — 채팅은 저장 안 하고 Broadcast로만 오가서 다시 볼 로그 자체가 없음 (`chat_messages` 테이블 항목 참고).
- 각 플레이어 말 위:
  - **금액 배지: 항상 표시** (말 색과 같은 테두리, `balance` 실시간 반영).
  - **채팅 말풍선: 채팅 시 잠깐(3.5초) 떴다 사라짐.** 길이 제한 20자.
  - 탈락자(파산 또는 자진 포기)는 흐리게 + "탈락" 표시.
- 좌우 코너(출발/무인도)는 아이콘 표시.
- 내 턴이 아니면 굴리기/구매 버튼 대신 "OO님 차례입니다" 문구만 보임 (다른 사람 턴에 조작 못 하게).
- 화면 하단에 "게임 포기하기" 버튼(탈락 전까지 항상 노출, 내 턴 여부 무관) — 확인창 한 번 거친 뒤 자진 탈락.

> 목업은 방향 확인용일 뿐. 색/간격/말풍선 스타일 등 세부는 구현하며 자유롭게 다듬어도 됨.

## 데이터베이스 스키마 (2단계에서 Supabase에 생성)

> 성격 구분: **신원**(사람) / **게임상태**(이번 판 한정) / **외형**(표시용 꾸미기).

### `rooms`
`id` uuid PK · `code` text · `status` text(waiting/playing/finished) · `turn_time_sec` int · `current_player_id` uuid · `turn_number` int · `turn_started_at` timestamp · `host_client_id` text · `created_at` timestamp

2단계 상태 동기화/자동진행을 위해 추가된 컬럼(엔진의 `GameState`를 그대로 담기 위함, `src/game-sync/mapping.ts` 참고):
`phase` text(awaiting-roll/awaiting-purchase-decision/**awaiting-build-decision**/**awaiting-initial-build-decision**/**awaiting-start-bonus-build**/**awaiting-space-travel-destination**/game-over) · `last_roll_d1` int · `last_roll_d2` int · `is_double_roll` bool · `pending_purchase_tile_idx` int(구매 대기·재방문 건축 대기·최초구매 즉시건축 대기가 이 컬럼을 공용으로 씀 — phase로 구분. 출발칸 보너스는 대상 칸을 플레이어가 그때그때 고르므로 이 컬럼을 안 씀, `DECIDE_START_BONUS_BUILD` 액션 자체에 `tileIdx`를 실어 보냄) · `winner_player_id` uuid · `notice` text · `version` int (낙관적 동시성 가드, 액션마다 1씩 증가)

`awaiting-build-decision`(재방문, 한 번에 한 단계 Y/N) / `awaiting-initial-build-decision`(최초구매 직후, 감당되는 등급 중 목표를 한 번에 선택 — 액션 `DECIDE_INITIAL_BUILD{targetLevel}`) / `awaiting-start-bonus-build`(출발칸 정확 도착, 보유 땅 중 하나 선택)는 3단계 건물 시스템에서 추가됨 — 새 컬럼 없이 기존 컬럼 재사용.

`awaiting-space-travel-destination`(우주여행 칸 도착 → 목적지 선택, 액션 `DECIDE_SPACE_TRAVEL{tileIdx}`)도 새 컬럼 없이 기존 컬럼만으로 구현 완료 — 대상 칸을 그때그때 고르는 방식이라 출발칸 보너스와 마찬가지로 `pending_purchase_tile_idx`를 안 쓰고 액션 자체에 `tileIdx`를 실어 보냄.

`event_deck` jsonb (**구현 완료** — 황금열쇠 덱의 남은 카드 순서. 셔플 결과를 서버가 들고 있어야 클라이언트가 다음 카드를 미리 못 봄. 3단계에서 처음으로 기존 컬럼 재사용이 아니라 진짜 새 컬럼을 추가한 사례 — 사용자가 SQL Editor에서 직접 마이그레이션 실행) · `welfare_pool` int (**구현 완료** — 사회복지기금 적립액, 같은 방식으로 마이그레이션) · `consecutive_doubles` int (**구현 완료** — 더블 3연속 → 무인도 판정용, 턴이 실제로 다음 사람에게 넘어갈 때 0으로 리셋, 같은 방식으로 마이그레이션).

| 컬럼 | 타입 | 성격 |

| id | uuid PK | |

- `appearance`를 jsonb로 두는 이유: 커스텀 항목이 늘어도 DB 스키마 변경 불필요.

- 확장 메모: 나중에 로그인 + 캐릭터 재사용 단계에서 신원+외형을 별도 `profiles` 테이블로 분리하고 `players`는 게임상태만 남기도록 리팩터링 예정.

### `tiles` (40칸 보드 정의, 고정 마스터 데이터)
`idx` int PK(0~39) · `name` text · `type` text · `price` int · `toll` int · `color_group` text
> DB에 생성 및 시드는 끝났지만(`src/game/board.ts`·`config.ts` 공식과 동일한 값), **클라이언트는 아직 이 테이블을 안 읽는다** — `engine.ts`가 여전히 로컬 상수 `BOARD`(`src/game/board.ts`)를 직접 참조. 지금은 둘이 같은 공식으로 만들어져 값이 같지만, 나중에 클라이언트가 DB에서 직접 읽도록 바꾸는 게 완전한 마이그레이션.
> 3단계: 등급별 통행료·건설비는 컬럼을 5쌍 늘리지 말고 **`price` 기준 배율 공식**(위 건물 시스템 표)으로 계산할 것. 배율 상수는 `config.ts` 한 곳에.

### `ownerships`
`id` uuid PK · `room_id` uuid FK · `tile_idx` int FK · `player_id` uuid FK · `level` int(프로토타입 0, 건물은 P1)
`unique(room_id, tile_idx)` 제약으로 같은 칸 중복 구매 방지.
> 3단계: `level`을 실제로 사용 — 0=땅만 / 1=별장1 / 2=별장2 / 3=빌딩 / 4=호텔. **재방문 업그레이드 + 최초구매 즉시건축 + 출발칸 정확 도착 보너스 셋 다 구현 완료**(`src/game/buildings.ts`의 공용 계산 + `gameReducer`의 `DECIDE_BUILD`/`DECIDE_START_BONUS_BUILD` 액션). 파산 시 건물 자동매각은 아직. 건물유지비 카드가 "플레이어별 등급별 건물 개수"를 세야 하므로, 이 테이블을 player_id로 집계하면 되도록 유지.

### `chat_messages` (선택)
`id` uuid PK · `room_id` uuid FK · `player_id` uuid FK · `body` text · `created_at` timestamp
> 실시간 채팅은 Broadcast로 즉시 전파가 기본. 이 테이블은 로그 다시보기 저장용. 초기엔 생략 가능.
> 2단계 채팅 슬라이스에서 실제로 생략함 — 현재 채팅은 Broadcast(`room:{roomId}:chat` 채널)로만 오가고 이 테이블엔 저장하지 않는다. 그래서 로비→게임 전환/새로고침 시 이전 대화가 사라진다. 나중에 로그 다시보기가 필요해지면 메시지 전송 시 이 테이블에 insert 한 줄 추가 + "로그 열기" UI만 얹으면 된다.

## 보안 (2단계에서 완료)

- **규칙 계산은 이제 전부 `supabase/functions/game-action`(Deno Edge Function, 서비스 롤 키)이 담당한다.** `ROLL_DICE` / `DECIDE_PURCHASE` / `CLAIM_TIMEOUT`(턴 타이머 자동진행) / `FORFEIT`(포기하기) 네 가지 액션 전부 이 함수 하나로 들어옴. 함수 안에서 `src/game/engine.ts`의 `gameReducer`를 그대로 실행(코드 중복 없이, 상대경로 import로 재사용 — Deno라 내부 import에 `.ts` 확장자가 필요해서 `engine.ts`/`board.ts`/`config.ts`/`types.ts`/`game-sync/mapping.ts`/`lobby/types.ts`만 그렇게 고쳐둠)한 뒤, 결과를 `rooms.version` 낙관적 락과 함께 DB에 반영한다.
  - 요청마다 서버가 직접 검증: `START_GAME`은 요청한 client_id가 `host_client_id`와 같은지, `ROLL_DICE`/`DECIDE_PURCHASE`는 요청한 client_id가 지금 턴인 플레이어와 같은지. 클라이언트가 브라우저 콘솔에서 이 검증을 우회할 방법이 없음(단, client_id 자체를 위조하는 것까진 못 막음 — 로그인이 없어서 생기는 한계, 아래 참고).
- **RLS + 컬럼 단위 권한으로 실제로 잠금.** `rooms`/`players`/`ownerships`에 RLS 활성화:
  - 읽기(select)는 anon 전부 허용 (로비/게임 화면 렌더링 + Postgres Changes 구독용).
  - `ownerships`는 anon용 insert/update/delete 정책이 아예 없음 — Edge Function(서비스 롤, RLS 항상 우회)만 쓸 수 있음.
  - `rooms`/`players`는 로비 CRUD(방 생성/참가/나가기)를 위해 insert/delete/update 정책은 있지만, **실제로 어느 컬럼을 고칠 수 있는지는 RLS가 아니라 컬럼 단위 `GRANT`로 잠갔다**: `rooms`는 `turn_time_sec`/`host_client_id`만, `players`는 `is_ready`만 anon이 직접 수정 가능. `balance`/`position`/`phase`/`version` 같은 게임 상태 컬럼은 anon 키로는 UPDATE 자체가 거부됨(Edge Function만 가능).
- **한계**: 로그인이 없어서 client_id 위조까진 못 막는다 — "내가 다른 사람인 척 요청을 보낸다"는 시나리오는 여전히 가능. 이건 CLAUDE.md가 애초에 말한 "부정행위(계산 조작)"보다 더 깊은 문제라 로그인 도입 전까진 범위 밖으로 남겨둠.
- 개발/배포: Supabase CLI 설치·로그인·프로젝트 링크 완료(`supabase link --project-ref qtfzhpafpsbnhjpkwdjj`). 배포는 Docker 없이 `supabase functions deploy game-action --use-api`로 잘 됨(이 프로젝트 규모에선 Docker 불필요했음).

## 개발 로드맵

- **1단계 (완료)** — 로컬 패스앤플레이. 40칸 보드, 주사위/이동/구매/통행료/파산/승리. 한 화면에서 턴 넘기며 규칙 검증.
- **2단계 (완료)** — Supabase 멀티플레이: 로비(방 생성/참가, Presence), 채팅(Broadcast), 상태 동기화(Postgres Changes), client_id 신원/재접속, 자동 진행(턴 타이머), 규칙 계산 Edge Function 이전, RLS+컬럼 권한 잠금. 추가로 게임 포기하기 기능도 이 단계에서 구현.
- **3단계 (P1, 현재)** — 재미 요소. 원작 부루마불 룰을 방치형에 맞게 각색: 건물 4등급 → 황금열쇠 → 사회복지기금(기존 "세금" 대체) → 더블 3연속/무인도 확장 → 독점 보너스(채택 미정).
- **4단계 (P2)** — Electron 포장(투명·항상위·작은 창, 트레이, 내 턴 알림), 캐릭터 외형 커스텀 UI(`appearance` 활용).

## 미정 사항 (구현하며 결정)

- 밸런스 수치 — 1단계에서 초기값 확정: 초기자본 1500 / 월급 200 / 땅값은 100부터 20씩 증가 (`src/game/config.ts`). 플레이 시뮬레이션 결과 2인전에서는 시간이 지날수록 월급 누적으로 파산이 잘 안 나 게임이 길어지는 경향 확인.
  - **3단계에서 적용 완료**: `LAND_TOLL_RATIO`를 20%→10%로 낮추고 건물 시스템(레벨별 통행료 배율 ×0.1~×4.5)을 도입함. 아직 실제 여러 판 플레이하며 재시뮬레이션은 안 해봄 — 플레이해보고 배율 조정 필요할 수 있음.
- 건물/땅 매각가 — 100%(`BUILDING_SALE_RATIO`/`LAND_SALE_RATIO`)로 구현 완료. 실제 여러 판 플레이해보고 파산이 너무 안 나면 50%로 낮출 것.
- 출발칸 정확 도착 보너스의 강도 — 1등급 상승이 적당한지, 아니면 건설비 할인/무료 같은 다른 형태가 나은지 플레이 후 판단.
- `landmark`(건물 못 짓는 고가 특수 땅, 원작의 서울 포지션) 도입 여부와 위치.
- 독점 보너스(통행료 2배) vs 독점 즉시 승리 vs 미도입 — 건물 시스템 플레이 후 결정.
- 승리 조건 — 1단계에서 "최후 1인 생존"으로 확정 구현. 턴 수 제한 후 자산 최고자 방식은 P1 이후 옵션으로 남겨둠.
- 로비→게임 창 전환 방식(Electron 단계): 같은 창 리사이즈 vs 별도 창. (2단계 이후 결정)
