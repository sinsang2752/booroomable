## 지금 작업 단계 (여기를 항상 먼저 확인)

> **현재: 2단계 완료, 3단계(P1)는 독점 보너스(항목 6)만 미정으로 남고 나머지 전부 완료 → 그 사이 4단계(P2) Electron 패키징을 먼저 진행해 상당 부분 완료됨**
>
> 로비, 채팅, 상태 동기화, client_id 신원/재접속, 자동 진행(턴 타이머), 규칙 계산 Edge Function 이전까지 2단계 전 항목 완료. 규칙 검증(누구 턴인지, 방장인지 등)과 실제 DB 쓰기는 이제 클라이언트가 아니라 `supabase/functions/game-action`(서비스 롤 키)이 전담하고, RLS+컬럼 권한으로 게임 상태 컬럼은 anon 키로 직접 못 쓰게 잠갔다.
>
> 3단계 마지막 항목(독점 보너스)이 "건물 시스템 플레이해보고 결정"인 채로 대기 상태라, 로드맵 순서를 어기고 4단계(Electron 패키징 + 데스크톱 위젯화)를 먼저 상당 부분 진행했다 — 자세한 내용은 아래 "Electron 패키징(데스크톱 위젯)" 절 참고. 지금은 두 단계가 병행 진행 중이라고 보면 됨.

**3단계 구현 순서 (한 슬라이스씩, 각 슬라이스마다 플레이 가능한 상태 유지)**

1. **건물 시스템** — 나머지 룰이 전부 건물 수/등급에 의존해서 가장 먼저 구현함.
   - **구현 완료**: 자기 땅 재방문 시 한 등급씩 업그레이드(레벨 0~4, 통행료/건설비 배율은 아래 표 그대로) + **최초 구매 시 감당되는 등급 중 목표를 한 번에 선택해 즉시건축**(`awaiting-initial-build-decision` phase) + **출발칸 정확 도착 보너스**(`awaiting-start-bonus-build` phase, 보드에서 업그레이드할 땅을 직접 클릭해서 고름). 세 흐름이 공유하는 "업그레이드 가능한가/비용" 계산은 `src/game/buildings.ts`에 순수 함수로 모아둠.
   - 파산 방지용 건물/땅 자동매각도 구현 완료 (아래 "매각 & 파산" 참고).
   - `landmark` 칸 — **구현 완료**(기존 "건너뛰기로 결정"에서 뒤집혀 도입 확정, 이후 실제 구현까지 끝남). 원작 재분석 결과 각 구역마다 건물 못 짓는 땅이 1~2개씩 섞여 있는 게 경제 구조의 핵심이라 6칸 도입. 상세는 "보드 칸 구성"과 "landmark" 절 참고.
2. **황금열쇠** — **1차 + 2차 전부 구현 완료** (아래 "황금열쇠" 참고). 3단계 로드맵 항목 중 완전히 끝난 두 번째 항목.
3. **사회복지기금** — **구현 완료** (아래 "사회복지기금" 참고).
4. **우주여행** — **구현 완료** (아래 "우주여행" 참고).
5. **더블 3연속 → 무인도**, 무인도 규칙 확장 — **구현 완료** (아래 "무인도" 참고).
6. **독점 보너스 / 승리 조건 추가** (아래 "독점 보너스" 참고 — 채택 여부 미정, 현재 미도입 유력) — 배율 상향 검증 후 판단.

**경제 구조 재설계 진행 상황** (원작 재분석 결과, 밸런스를 개별 숫자가 아니라 구조 단위로 재설계):

1. **땅 이름 → 실제 지역명** (`board.ts` + DB `tiles.name`) — `board.ts` 코드는 완료. DB `tiles.name` UPDATE SQL은 `supabase/schema.sql` 맨 아래에 있고 사용자가 SQL Editor에서 직접 실행하는 부분만 남음(게임 로직 자체엔 영향 없음 — `tiles` 테이블은 클라이언트가 안 읽음).
2. **초기자본 1500 → 3000** — **구현 완료**(`config.ts`). 월급 200은 그대로.
3. **땅값을 구역별 계단식으로 교체** — **구현 완료**(`board.ts`의 `LAND_DEFS`, "보드 칸 구성" 표 그대로).
4. **건설비를 땅값 비례 → 구역별 고정 절대값으로** — **구현 완료**(`config.ts`의 `ZONE_BUILD_COST`, `buildings.ts::getUpgradeCost`가 유일한 계산 지점).
5. **`landmark` 타입 구현** — **구현 완료**. 건물 관련 세 흐름(최초구매 즉시건축/재방문 업그레이드/출발칸 보너스) 전부에서 `getUpgradeCost`가 `null`을 리턴하는 것 하나로 자동 제외되고, 통행료는 `board.ts`에 미리 넣어둔 고정값이 그대로 나간다(레벨이 영원히 0이라 배율 곱셈이 항등식이 되는 걸 이용 — 별도 분기 코드 없이 `resolveTileLanding`의 `empty_land` 분기에 `landmark`만 조건 추가해서 해결).
   - 구현 중 발견한 버그: `src/components/TurnPanel.tsx`가 `buildings.ts::getUpgradeCost`를 거치지 않고 건설비 공식을 **독자적으로 중복 구현**하고 있었음 — 고쳐서 하나의 계산 지점으로 통일함(안 고쳤으면 재방문 업그레이드 화면에 옛날 공식 기반의 틀린 금액이 떴을 것).
6. **통행료 배율 상향** (빌딩 ×3→×4, 호텔 ×4.5→×8) — **구현 완료**(`config.ts`의 `BUILDING_TOLL_LEVEL_MULTIPLIERS`).
7. **재시뮬레이션으로 측정**(2인전/4인전 각 50판, 평균 턴 수·파산율·평균 최종 자산) — **완료**. 결과와 방법론은 "미정 사항"의 "밸런스 수치" 항목 참고.

> 2~6번은 서로 얽혀 있어서 하나의 슬라이스로 같이 반영함(초기자본만 올리고 땅값은 그대로면 오히려 더 안 끝나서). 골든키 카드 금액(상금/벌금/유지비)은 이번에 재스케일 안 함 — 초기자본이 두 배 됐으니 상대적으로 카드 효과가 약해진 상태로, 다음 밸런스 패스에서 같이 볼 것.

**4단계(P2) 진행 상황** — 위 미정 항목을 미루고 Electron 패키징을 먼저 진행함. 포터블 zip 빌드/투명·항상위·우측하단 위젯 창/화면별 동적 창 크기/반응형 폰트 크기/커스텀 폰트(Galmuri11 Bold + Noto Sans KR 이원화, 이전 Dongle 대체) 적용까지 **구현 완료**. 자세한 내용은 아래 "Electron 패키징(데스크톱 위젯)" 절 참고. 트레이 아이콘·내 턴 알림·캐릭터 외형 커스텀은 아직.

**이번 단계에서 구현하지 않기로 확정한 것** (원작에 있지만 방치형과 안 맞아 제외):
자기 턴에 아무 땅에나 상시 건물 짓기(단 **출발칸 정확 도착 시 1등급 보너스는 예외로 도입** — "건물 시스템" 참고) / 우대권 / 무인도 탈출권(무전기) / 대출·담보대출.

**자동 기본값의 대원칙**: 타임아웃으로 자동진행되면 **아무것도 하지 않고 턴을 넘긴다.** 구매·건축·우주여행 이동 전부 동일. 방치한 쪽이 이득을 보는 상황이 생기지 않도록 예외를 두지 않는다.

단계가 끝나면 이 섹션을 다음 단계로 갱신할 것. 전체 로드맵은 맨 아래 "개발 로드맵" 참고.

## 항상 지켜야 할 규칙

1. **한 번에 하나씩, 돌아가는 것부터.** 큰 기능을 통째로 만들지 말고, 작게 쪼개서 각 단계마다 실행되는 상태를 유지한다. 게임 개발 경험이 없는 개발자와 함께 작업 중임을 감안해 설명을 곁들인다.
2. **상태의 단일 출처(single source of truth).** 2단계 이후 게임의 진짜 상태는 항상 서버(Supabase DB)에 있고 클라이언트는 그것을 구독/표시할 뿐이다. 1단계에서는 로컬 상태로 시작하되, 나중에 서버로 옮기기 쉽게 게임 로직과 UI를 분리해서 짠다.
3. **밸런스 수치는 하드코딩하지 말고 한 곳에 모은다.** 초기 자본, 칸별 가격/통행료, 월급액 등은 설정 파일/상수 한 군데(예: `src/game/config.ts`, 보드 정의는 `src/game/board.ts`)에 모아 플레이하며 쉽게 조정할 수 있게 한다.
4. **모든 선택지에는 "자동 기본값"이 반드시 정의되어야 한다.** 이 게임은 방치형이라 턴 제한시간이 지나면 시스템이 대신 결정한다. 새 룰을 추가할 때 플레이어의 판단이 필요한 지점이 생기면, 그 지점의 자동 동작을 반드시 같이 정의할 것. 자동 기본값을 정할 수 없을 만큼 복잡한 선택이라면 그 룰 자체가 이 게임에 안 맞는다고 보고 단순화한다. (원작 부루마불 룰 중 상당수가 여기에 걸려서 각색이 필요함)
5. **큰 작업은 Plan 모드로 계획을 먼저 보여준다.**
6. 큰 변경 전에는 무엇을 왜 바꾸는지 한국어로 짧게 설명한다.

## 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 빌드/프론트 | Vite + React + TypeScript | 1단계부터 사용 |
| 백엔드/DB | Supabase (Postgres) | 2단계부터 |
| 실시간 로비 | Supabase Realtime Presence | 접속자 목록 |
| 실시간 채팅 | Supabase Realtime Broadcast | 방 채팅 |
| 게임 상태 동기화 | Supabase Postgres Changes | DB 변경 구독 |
| 부정행위 방지 | Supabase Edge Function | 완료 — `supabase/functions/game-action` |
| 데스크톱 포장 | Electron | 4단계에서 상당 부분 구현 완료 — 아래 "Electron 패키징" 참고 |

> 턴제라 밀리초 동기화가 불필요 → Socket.io는 쓰지 않고 Supabase Realtime으로 처리.

## 게임 규칙

### 보드
- **40칸 타원형(알약형) 트랙**: 위 19칸 + 아래 19칸 + 좌우 코너 각 1칸.
  - 좌 코너 = 출발, 우 코너 = 무인도.
- 데스크톱 하단 배치를 고려해 가로로 넓고 세로로 얇은 형태.
- 칸 종류
  - 2단계까지: `empty_land`(빈 땅) / 소유된 땅 / `start`(출발) / `jail`(무인도).
  - 3단계 추가: `event`(황금열쇠, **구현 완료** — `EVENT_TILE_INDICES`) / `welfare_pay`·`welfare_get`(사회복지기금, **구현 완료** — `WELFARE_PAY_TILE_IDX`/`WELFARE_GET_TILE_IDX`) / `space_travel`(우주여행, **구현 완료** — `SPACE_TRAVEL_TILE_IDX`) / `landmark`(건물 못 짓는 고가 특수 땅, 도입 보류).
- **특수칸 배치 주의**: 원작은 정사각형이라 네 모서리에 출발·무인도·기금수령·우주여행을 배치하지만, 우리 보드는 코너가 좌우 2개(출발·무인도)뿐이다. 나머지 특수칸은 위/아래 줄 안에 배치해야 하며, 한쪽에 몰리지 않게 **황금열쇠 칸은 트랙 전체에 고르게 분산**시킬 것(원작도 각 구역마다 배치되어 있음).

### 보드 칸 구성 (40칸 확정 — `src/game/board.ts`)

특수칸 9개(출발·무인도·우주여행·기금납부·기금수령·황금열쇠 4) + **땅 31칸**(건물 가능 25 + `landmark` 6).

**환산 기준: 원작 1만 원 = 우리 10.** 원작 월급 20만 원 → 우리 200으로 지금 값과 정확히 일치해서, 월급은 그대로 두고 나머지를 원작 비율에 맞췄다.

**땅값은 등차수열이 아니라 구역별 계단식**이다. 원작이 구역(라인)마다 가격대를 층으로 나누고 같은 구역 안에서는 비슷한 값을 반복하는 구조라서 그대로 따랐다. 등차수열은 인접 칸 차이가 미미해 "이 땅이 특별하다"는 감각이 없다.

**특수칸**: 0 출발 / 5·15·25·35 황금열쇠 / 10 기금납부 / 18 우주여행 / 20 무인도 / 30 기금수령 — **인덱스 전부 기존 그대로**. 이름·가격만 바뀌므로 인덱스를 참조하는 로직(황금열쇠 이동 카드 등)은 건드릴 필요 없다.

**컬러그룹**: Zone 1~4를 그대로 컬러그룹으로 쓴다(기존 6그룹 → 4그룹). 원작의 라인 개념과 일치하고, 건설비가 구역 단위로 묶이므로 색과 건설비가 시각적으로 대응된다.

### 건설비: 구역별 고정 (원작 구조 반영, 기존 "땅값 비례"에서 변경)

**원작은 건설비가 땅값에 비례하지 않고 구역마다 고정이다.** 1구역은 어느 도시든 별장 5만/빌딩 15만/호텔 25만, 4구역은 별장 20만/빌딩 60만/호텔 100만 식. 그래서 **같은 구역 안에서 비싼 땅일수록 투자 효율이 좋아지고**, "이 구역에선 이 땅을 노려야 한다"는 판단이 생긴다. 나무위키가 이스탄불(1구역 최고가)을 "건물·호텔 가격대비 수익률 최고", 오타와(2구역 최고가)를 "빌딩 가격대비 수익률 200%로 최고"로 평가하는 게 정확히 이 구조 때문이다.

기존의 "건설비 = 땅값 × 배율" 방식은 모든 땅의 ROI가 동일해서 전략적 선택이 아예 없었다. → **구역별 고정 절대값으로 변경**(위 표).

- 별장2는 별장1과 같은 금액을 한 번 더 낸다(원작 동일).
- 레벨을 건너뛰어 올릴 때(최초 구매 즉시건축)는 거쳐 가는 단계의 비용을 전부 합산해서 낸다. 예: Zone 4 땅을 사자마자 호텔까지 = 땅값 + 200 + 200 + 600 + 1000.

### `landmark` — 건물을 지을 수 없는 땅 (도입 확정, 기존 "보류"에서 변경)

원작에는 각 구역마다 건물 건설이 불가능한 대신 통행료가 고정으로 높은 땅이 1~2개씩 섞여 있다(제주도·부산·서울·탈것들). 건물을 못 지으니 추가 투자가 필요 없고, 사두면 그 즉시 제 역할을 한다.

- **통행료 = 땅값의 약 1.4배로 통일**. 원작은 이 비율이 제각각인데(제주 1.5배 / 부산 1.2배 / 서울 2.0배), 퀸 엘리자베스호 0.83배·컬럼비아호 0.89배처럼 **통행료가 땅값보다 싼 설계 실수**가 있어서 그대로 쓰지 않았다. 우리는 전부 1.4배 근처로 맞춰 "사면 손해인 땅"이 없게 했다.
- **서울만 1.56배**로 약간 높다. 원작의 상징이라 최강 땅 지위는 유지하되, 원작처럼 압도적이진 않게 조정했다. 원작 서울(통행료가 초기자본의 68%)은 밸런스 파괴로 유명한데, 우리 서울은 1400으로 초기자본(3000)의 47% 수준이다.
- **후반에는 건물 올린 땅이 서울을 넘어선다**: 뉴욕/런던(땅값 350)에 호텔을 올리면 통행료 2800으로 서울 1400의 두 배. 원작에서 이상적이라고 평가되는 구도(서울이 초반을 지배하고 후반엔 개발된 대도시가 역전)를 의도적으로 만든 것.
- **역할 분담**: `landmark`는 저위험·즉시 효과·수익 상한 있음 / 건물 가능 땅은 고위험·느린 시작·상한 없음. 어느 쪽에 투자할지가 전략이 된다.
- 건물 관련 UI/로직에서 `landmark`는 전부 제외 대상 — 최초구매 즉시건축, 재방문 업그레이드, 출발칸 보너스 선택지 모두에서 빠진다(이미 "건물 시스템"에 제외 규정이 있으니 그대로 적용).

### 인원
- **최소 2명 ~ 최대 4명.** 2명부터 시작 가능. (정확히 4명이 차야 하는 게 아님)
- 말 색은 입장 순서대로 자동 배정: blue → red → green → amber.

### 이동 & 턴
- 주사위 2개. 더블(같은 눈)이면 한 번 더 굴림.
- 출발 칸을 지나가면 월급 지급. **출발 칸에 정확히 멈추면** 월급에 더해 자유 건축 보너스가 발동한다("건물 시스템" 참고) — 지나가는 것과 정확히 멈추는 것을 구분해서 처리할 것.
- 주사위는 이제 서버(`game-action` Edge Function)에서 생성. 브라우저는 "굴려줘" 요청만 보내고, 실제 `Math.random` 호출과 결과 계산은 Deno 서버 안에서 일어난다.

### 땅 & 통행료
- 빈 땅 도착 → 구매/패스 선택. 단, **땅값보다 잔액이 적으면 애초에 물어보지 않고 그냥 지나감** (살 수도 없는데 선택지를 보여줄 필요가 없어서, 2단계에서 추가).
- 남의 땅 도착 → 통행료 자동 징수.
- 통행료는 소유 등급(건물)에 따라 결정. 아래 "건물 시스템" 참고.

### 건물 시스템 (3단계, 원작 부루마불 기준으로 각색)

**등급**: `level` 0~4 — 0=땅만(대지료) / 1=별장1 / 2=별장2 / 3=빌딩 / 4=호텔.
- 원작대로 한 땅에 건물은 한 채까지이며, 별장만 예외적으로 2채가 가능하다. 그래서 별장1·별장2를 별도 등급으로 둔다.
- 상위 등급을 지으면 하위 건물은 대체된다(철거·재건축 정산 없음 — 단순화).

**언제 지을 수 있나 — 도착 시에만 (확정. 원작 정식룰과 다름, 의도된 각색)**
- 원작 정식룰은 자기 턴이면 말 위치와 무관하게 자기 땅 아무 곳에나 건물을 지을 수 있고, 나무위키는 "도착한 땅에만 짓기"를 게임을 지루하게 만든 변형룰이라고 비판한다. 하지만 이 프로젝트는 **방치형**이라 "내 땅 여러 곳 중 어디에 무엇을 지을지"를 매 턴 고르게 만들 수 없고, 자동 기본값도 정하기 어렵다. → **자기 소유 땅에 도착했을 때만 그 땅을 업그레이드**하는 방식을 채택한다. **매 턴 자유 건축은 채용하지 않는다 — 확정.** (단 출발칸 정확 도착 시에는 예외적으로 자유 건축 1회 허용. 아래 참고)
- 대신 원작 변형룰의 "게임이 늘어진다"는 부작용은 아래 두 가지로 상쇄한다:
  - **최초 구매 시 자금이 허용하는 범위에서 건물까지 한 번에 지을 수 있다** (모바일 모두의마블의 '신의 손' 계열 룰). 땅만 사고 4번을 더 방문해야 호텔이 되는 구조보다 훨씬 빠르다. **UI는 한 단계씩 반복해서 묻지 않고, 감당 가능한 등급(별장1/별장2/빌딩/호텔) 중 원하는 목표를 한 번에 선택하는 방식**(`awaiting-initial-build-decision` phase, 액션 `DECIDE_INITIAL_BUILD{targetLevel}`) — 서버가 `targetLevel`을 그대로 안 믿고 감당 가능한 최고 레벨로 다시 클램프한다.
  - 재방문 시에는 **한 번에 한 등급씩** 상승(Y/N만 묻는 기존 방식 그대로, `awaiting-build-decision`).
- **자동 기본값**: 자동진행으로 넘어간 경우 건물은 짓지 않는다(패스). 구매도 현행대로 자동 패스.

**예외 — 출발칸 정확 도착 보너스 (구현 완료)**
주사위 눈이 딱 맞아 **출발 칸에 정확히 멈춘 경우에 한해, 자기 소유 땅 중 아무 곳이나 하나를 골라 한 등급 올릴 수 있다.** (모바일 모두의마블의 출발지 보너스 규칙. 원작 부루마불에도 특수능력 변형룰로 소개되어 있음)
- 상시 자유 건축은 방치형과 안 맞아 채용하지 않지만, **출발칸 정확 도착이라는 낮은 빈도로 제한하면** 부담 없이 전략성만 챙길 수 있다. 한 바퀴에 한 번 올까 말까 한 기회라 "이번엔 어디에 지을까"를 고민할 만한 순간이 된다.
- 세부 규칙:
  - **정확히 멈춰야** 한다. 지나가기만 한 경우는 해당 없음(월급은 지나가도 받지만 이 보너스는 별개).
  - **한 등급만** 상승. 여러 채를 한 번에 짓거나 여러 땅에 나눠 짓는 것은 불가.
  - 건설비는 정상 지불. 자금이 부족하면 그 등급은 선택할 수 없다.
  - 대상에서 제외: 이미 호텔(level 4)인 땅, `landmark`(건물 못 짓는 땅), 소유하지 않은 땅.
  - 지을 수 있는 땅이 하나도 없으면 선택 단계를 건너뛴다.
- **자동 기본값**: 제한시간 내에 고르지 않으면 **아무것도 짓지 않고 그대로 턴을 넘긴다.**
- UI: 보드에서 업그레이드 가능한 자기 땅을 하이라이트(테두리 펄스 효과)하고, 칸에 마우스를 올리면 다음 등급/건설비가 툴팁으로 표시됨. "건너뛰기" 버튼도 제공.
- `landmark` 칸은 아직 미도입이라 현재 "제외 대상"은 실질적으로 이미 호텔인 땅/소유하지 않은 땅 두 가지만 적용됨.

**통행료 배율 (원작 실측 비율 기반, `config.ts`에 상수로)**

통행료는 **건물 가능 땅에만** 적용되는 배율 공식이다(`landmark`는 고정 통행료 — "보드 칸 구성" 참고).

| level | 건물 | 통행료 (땅값 대비) |
|---|---|---|
| 0 | 없음 | ×0.1 |
| 1 | 별장1 | ×0.5 |
| 2 | 별장2 | ×1.5 |
| 3 | 빌딩 | **×4** |
| 4 | 호텔 | **×8** |

> **건설비는 이 표에 없다** — 땅값 비례에서 **구역별 고정 절대값**으로 바뀌었다. "건설비: 구역별 고정" 절 참고.
> **배율 상향 근거 (원작 재분석 결과)**: 원작 부루마불은 우리와 달리 **건물이 대체되지 않고 공존하며 통행료가 합산**된다(별장1+별장2+빌딩+호텔 전부 한 땅에). 나무위키 각주로 검증됨 — 타이베이 풀하우스 통행료 38만 = 대지료 0.2 + 별장1 1 + 별장2 3 + 빌딩 9 + 호텔 25. 뉴욕 풀하우스 자산가치 235만(= 땅값 35 + 별장 20×2 + 빌딩 60 + 호텔 100)의 절반이 반액대매출 117.5만인 것도 일치.
> 원작의 누적 통행료를 땅값 대비로 환산하면 **최종 단계가 땅값의 7.6배(타이베이)~9.4배(런던)**. 기존 ×4.5는 이보다 훨씬 완만해서 "파산이 안 나 게임이 길어진다"는 문제의 원인이었다.
> **우리는 건물 합산 구조를 채택하지 않는다(단일 등급 대체 유지 — UI·구현이 훨씬 단순)**. 대신 **최고 등급 배율을 원작의 누적 결과치에 맞춰 올려 같은 효과를 낸다.** 그래서 호텔 ×8, 빌딩 ×4.
> **구현 완료** — `config.ts`의 `BUILDING_TOLL_LEVEL_MULTIPLIERS`가 이 표 그대로 `[1, 5, 15, 40, 80]`.

(건물을 지을 수 없는 특수 땅은 위 "landmark" 절에서 이미 도입 확정 + 구현 완료로 정리됨 — 여기 있던 중복 서술은 제거.)

### 매각 & 파산 (구현 완료)
- 통행료를 낼 현금이 부족하면 **자동으로 낮은 등급 건물부터 매각**해 충당한다. 매각가는 건설비의 100%(`BUILDING_SALE_RATIO`, `config.ts`. 모노폴리처럼 반액으로 할지는 플레이해보고 밸런스 보고 결정).
- 건물을 다 팔아도 부족하면 **땅 매각**(`LAND_SALE_RATIO`, 땅값의 100%), 그래도 부족하면 파산.
- **자동 기본값**: 위 순서(낮은 등급 건물 → 높은 등급 건물 → 싼 땅 → 비싼 땅)로 시스템이 알아서 처분. 플레이어에게 무엇을 팔지 묻지 않는다(방치형이라). `src/game/buildings.ts`의 `liquidateForDebt` 순수 함수 + `engine.ts`의 통행료 분기에서 호출.
- 구현하면서 발견한 기존 버그: `computePatches`(`src/game-sync/mapping.ts`)가 원래 "액션 한 번에 칸 하나만 바뀐다"고 가정해 변경된 칸을 하나만 찾고 멈추는 구조였는데, 매각은 한 번에 여러 칸(건물 여러 채 + 땅 여러 곳)이 동시에 바뀔 수 있어 이번에 배열(`ownershipUpserts`/`ownershipDeletions`)로 고쳤다. 이 컬럼들이 없었으면 매각 결과 일부가 DB에 조용히 반영 안 될 뻔했음.
- **대출 시스템은 구현하지 않는다 (확정).** 원작에는 은행 대출(1회 한정, 3턴 내 상환) 룰이 있지만, 상환 판단의 자동 기본값을 정하기 어렵고 게임이 늘어진다. 담보대출 등 변형룰도 마찬가지로 전부 제외.

### 황금열쇠 (3단계)

`tiles.type = 'event'` 칸(`config.ts`의 `EVENT_TILE_INDICES = [5, 15, 25, 35]`, 트랙에 고르게 4곳)에 도착하면 카드 1장을 뽑아 즉시 효과 적용. 덱은 셔플 후 순환.

**1차 구현 — 전부 완료**
- **상금(구현 완료)**: 고정액 수령. 원작 예시(노벨상 30만/복권 20만/장학금 10만/연금 5만)를 우리 경제 규모(초기자본 1500)에 맞춰 1/1000 스케일 → 노벨상 300 / 복권 200 / 장학금 100 / 연금 50 (`config.ts`의 `GOLDEN_KEY_DECK`).
- **벌금(구현 완료)**: 고정액 지불(해외유학 100 / 병원비 50, 같은 스케일). 통행료와 똑같은 "빚"으로 취급해 잔액이 부족하면 건물 시스템의 자동매각(`liquidateForDebt`)이 먼저 발동하고, 그래도 부족하면 파산.
- **생일축하(구현 완료)**: 생존한 다른 모든 플레이어에게서 30씩 징수. 각자 가진 만큼만 내고(부족해도 파산까지는 안 감 — 문서에 부족분 규정이 없어 단순화) 현재 플레이어에게 합산 지급.
- **이동(구현 완료)**: 출발지로 이동(월급 받음 + 출발칸 정확 도착 보너스 체크까지 이어짐) / 무인도로 이동(월급 없음, 다음 턴 스킵) / 뒤로 2칸 · 뒤로 3칸 이동(월급 없음, 별개 카드 2장으로 "2~3칸"의 다양성을 자연스럽게 구현).
  - 원작 규칙 그대로: 이동 카드는 "목적지에 닿을 때까지 계속 전진"하는 개념이라, **이동한 칸에 도착하면 그 칸의 정상 효과(통행료 지불/구매 프롬프트/건물 업그레이드 프롬프트 등)까지 이어서 처리**한다.
  - 구현을 위해 `handleRollDice`에 있던 "칸 도착 효과" 로직을 `resolveTileLanding` 함수로 뽑아 리팩터링(`src/game/engine.ts`) — 주사위 이동과 황금열쇠 이동 카드 양쪽에서 재사용. 황금열쇠 이동 카드가 하필 다른 황금열쇠 칸에 떨어지는 극히 드문 경우엔 카드를 또 뽑지 않고 지나간다(재귀 방지, 문서에 규정 없는 단순화).
  - **`특정 칸으로 이동` 카드는 도입하지 않음(확정)** — 원작의 철도·전기회사처럼 우리 보드에 없는 "지정 대상"을 가리켜야 하는 카드라 뭘 가리킬지 불명확해서 스킵.
- 카드 덱은 `rooms.event_deck`(jsonb, 3단계에서 신규 추가된 컬럼)에 안 뽑은 카드 순서를 서버가 들고 있다가 셔플 후 순환(`src/game/engine.ts`의 `shuffleDeck`, 주사위와 동일하게 서버에서만 `Math.random()` 실행).

**2차 구현 — 전부 완료**
- **건물유지비 3종(구현 완료)**: 보유 건물 등급(별장=레벨1·2 공통/빌딩=레벨3/호텔=레벨4)별 단가 × 개수를 합산해 은행에 지불. 원작 비율을 1/1000 스케일 → 정기종합소득세(별장30/빌딩100/호텔150) > 건물수리비(30/60/100) > 방범비(10/30/50). 벌금과 동일하게 "빚"으로 취급해 부족하면 자동매각(`liquidateForDebt`) 후 그래도 부족하면 파산.
- **반액대매출(구현 완료)**: 보유한 땅 중 총 가치(땅값+누적 건설비, `src/game/buildings.ts`의 `getPropertyValue`)가 가장 큰 곳을 골라 그 절반을 현금으로 받고 소유권·건물을 통째로 잃는다(다시 "빈 땅"으로 돌아가 누구나 살 수 있음). 소유한 땅이 없으면 아무 일도 안 일어남.

**미도입 (확정)**
- **우대권**(통행료 1회 면제), **무인도 탈출권(무전기)** 은 구현하지 않는다. 보유했다가 원하는 때 쓰는 카드라 "언제 쓸지"가 방치형과 충돌하고, 보유 아이템 상태를 따로 관리해야 해서 복잡도만 늘어난다. → 황금열쇠 덱에서 제외.

### 우주여행 (구현 완료)

`type = 'space_travel'` 칸(idx18, `SPACE_TRAVEL_TILE_IDX`)에 도착하면 **보드의 원하는 칸으로 이동**한다. 원작에서 가장 전략적인 칸.

**진행 방식**
- 도착 즉시 이동지를 정한다(새 phase `awaiting-space-travel-destination`, 액션 `DECIDE_SPACE_TRAVEL{tileIdx}`).
- **우주여행 칸으로는 다시 갈 수 없다** — 목적지 후보에서 아예 제외(서버가 `tileIdx`로 우주여행 칸이나 현재 위치를 보내면 무시하고 "이동 안 함"과 동일하게 처리, 클라이언트를 신뢰하지 않음).
- 이동은 텔레포트가 아니라 **트랙을 따라 전진하는 판정**이다. `목적지 idx < 현재 위치`면 앞으로 걷다가 39→0으로 넘어간 것이므로 월급을 받는다(원작 규칙과 동일).
- 도착한 칸의 효과는 정상 적용 — 통행료/구매 프롬프트/건물 프롬프트/출발점 보너스/황금열쇠까지 전부 `resolveTileLanding` 공용 함수로 자연스럽게 이어짐(칸 도착 효과 로직을 함수로 뽑아둔 덕에 새 코드 없이 재사용).
- **이동 후에는 더블이어도 추가 굴림 없이 턴을 종료한다** — `isDoubleRoll`을 이동 시점에 강제로 `false`로 덮어써서, 도착한 칸이 또 구매/건축 프롬프트로 이어지는 경우까지 전부 재굴림 없이 처리되도록 보장(체인 도중 어디서 최종 `finishTurnStep`이 불려도 안전).
- 원작의 "컬럼비아호 소유자에게 이용료 지불"은 탈것 개념이 없으므로 **생략**.

**자동 기본값 (방치형 핵심)**
- 제한시간 내에 목적지를 고르지 않으면 **아무것도 하지 않고 그대로 턴을 넘긴다.**

**UI**
- 출발점 보너스 때 만든 보드 클릭 선택 메커니즘(`selectableTiles`/`onSelectTile`)을 그대로 재사용 — 목적지 후보 전부(우주여행 칸·현재 위치 제외) 하이라이트, 잔액으로 못 사는 빈 땅도 선택 가능(이동 자체는 안 막음).
- "이동 안 함" 버튼 제공.

### 사회복지기금 (구현 완료, 기존 "세금" 항목을 대체)
원작의 기부 칸 + 접수 칸 구조. 세금을 그냥 은행에 버리는 것보다 훨씬 재밌고 구현도 단순하다.
- `type = 'welfare_pay'` 칸(idx10): 도착 시 고정액(`WELFARE_PAY_AMOUNT`=100)을 **기금 풀에 적립**(`rooms.welfare_pool`). 통행료·황금열쇠 벌금과 동일하게 "빚"으로 취급해 잔액 부족하면 자동매각(`liquidateForDebt`) 후에도 부족하면 파산.
- `type = 'welfare_get'` 칸(idx30): 도착 시 **적립된 기금 전액 수령**. 비어 있으면 아무 일도 없음.
- 황금열쇠 덱에 "사회복지기금 접수처로 이동" 카드 추가 — 다른 이동 카드처럼 도착한 칸(항상 `welfare_get`)의 효과가 이어지고, 이동 중 출발점을 지나치면 월급도 받는다(일반 이동 카드 규칙과 동일, 무인도행만 예외).

### 무인도 (구현 완료)
- 원작 그대로 **3턴 대기**(`players.jail_turns_left`, 무인도 도착 시 3으로 설정). 예전 1턴 스킵용 `skip_next_turn` bool은 폐기 예정이었으나 컬럼 자체는 안전하게 남겨두고(DROP 안 함) 코드에서만 안 쓴다.
- **핵심 변경**: 무인도에 갇힌 플레이어도 이제 자기 턴마다 실제로 주사위를 굴린다(기존엔 턴 자체가 로테이션에서 통째로 건너뛰어져 굴릴 기회가 없었음). 더블이면 그 눈만큼 이동해 즉시 탈출, 아니면 이동 없이 대기 턴만 1 줄고, 3번째 시도에서도 더블이 안 나오면 그 굴림으로 강제 석방+이동한다(원작과 동일 — 대출이 없는 이 프로젝트 특성상 "3번째엔 벌금 내고 나가기" 대신 그냥 무료로 풀어줌). 탈출/석방 이동은 `resolveTileLanding`을 그대로 재사용해 도착 칸 효과(구매/건축/황금열쇠 등)까지 정상 적용하되, **보너스 재굴림은 절대 없음**(탈출에 쓴 더블이든 이후 정상 착지가 우연히 또 더블이었든 전부 `isDoubleRoll`을 강제로 꺼서 처리 — 안 그러면 방금 들어간 무인도를 같은 턴에 또 탈출 시도하는 버그가 생김. 구현 중 실제로 이 버그를 잡아서 고쳤음).
- 새 phase/액션 없이 기존 `ROLL_DICE` + `awaiting-roll`만으로 전부 표현됨 — 타임아웃 자동진행도 무변경으로 그대로 작동(무인도 중이어도 시간 초과 시 서버가 대신 `ROLL_DICE`를 제출).
- 원작에서 무인도는 후반부에 오히려 피난처가 된다(건물이 깔린 판을 안전하게 넘길 수 있어서). 이 성격은 그대로 살아있다.
- **더블 3연속 → 무인도 강제 이동 (구현 완료)** — `rooms.consecutive_doubles`로 추적, 턴이 실제로 다음 사람에게 넘어갈 때 0으로 리셋. 3번째 연속 더블이면 이동 계산 자체를 안 하고 바로 무인도 칸으로 텔레포트(`resolveTileLanding` 호출 안 함) — 그래서 "출발지를 지나쳐도 월급을 받지 못한다"는 요건이 코드 구조상 자동으로 충족됨.
- UI: 말 배지에 "무인도 N" 표시 추가(`PlayerToken.tsx`, 파산 배지와 같은 자리, 탈락자와 헷갈리지 않게 다른 색). 그 외 별도 프롬프트 없음 — notice 텍스트로 상황 설명.

### 독점 보너스 / 추가 승리 조건 (채택 여부 미정)
- 기존 로드맵의 "컬러그룹 독점 보너스(같은 색 전부 소유 시 통행료 2배)"는 **원작 부루마불에는 없는 모노폴리 룰**이다. 목표 배율(호텔 ×8)을 적용하면 통행료가 이미 땅값의 8배까지 오르므로, 여기에 독점 2배까지 겹치면 확실히 과하다. → 배율 상향분을 먼저 플레이로 검증한 뒤 판단할 것. 지금 시점에서는 **미도입이 유력**.
- 대안: 모두의마블식 **독점 즉시 승리** — 같은 컬러그룹을 전부 소유하면 그 자리에서 승리. 방치형 게임에 명확한 종료 조건을 하나 더 주므로 "게임이 길어진다" 문제에 오히려 잘 맞는다.
- 둘 다 넣지 않고 건물 시스템만으로 밸런스가 잡히면 그대로 두는 것도 정답.

### 자산 & 승패
- 초기 자본 균등 지급.
- 통행료를 못 내면 파산 → 탈락.
- **게임 도중 언제든 "게임 포기하기"로 자진 탈락 가능** (2단계에서 추가). 확인창 한 번 거친 뒤 파산과 동일하게 처리(`isBankrupt=true`, 잔액 0). 내 턴에 포기하면 턴이 다음 사람에게 넘어가고, 남의 턴에 포기해도 진행 중인 턴은 안 끊김.
- 최후 1인 생존 시 승리. (P1: 턴 수 제한 후 자산 최고자 승리 옵션)

### 자동 진행 (방치형 핵심, 2단계에서 구현 완료)
- 각 턴 제한시간: 기본 30초, 방 설정으로 20~60초 (`rooms.turn_time_sec`, 로비에서 방장이 설정).
- 시간 초과 시 자동으로 주사위를 굴림 (구매 대기 중이었다면 자동 패스).
- **자리를 비워도 멈추지 않는 설계** (`src/hooks/useSyncedGame.ts`):
  1. 매 행동(굴리기/구매결정, 수동이든 자동이든) 성공 시 `rooms.turn_started_at`을 항상 `now()`로 갱신 — "이번 대기가 언제 시작됐는지"를 나타냄.
  2. 접속한 아무 클라이언트나 2초마다 "현재 시각 − turn_started_at > turn_time_sec?"을 확인 — 이때 "현재 시각"은 `Date.now()`가 아니라 `src/lib/serverClock.ts`의 `getServerNow()`(PC 로컬 시계 오차를 서버 기준으로 보정한 값)를 쓴다. 로컬 시계가 어긋난 PC가 실제보다 이르게/늦게 시간 초과라고 판단하는 걸 줄이기 위함(최종 판정은 아래처럼 항상 서버가 다시 하므로 이건 클라이언트 쪽 판단 정확도 개선일 뿐).
  3. 초과 시 곧바로 자동 행동을 제출하지 않고, `rooms.version`을 조건부 업데이트(`WHERE version = 관찰한 값`)로 먼저 선점 — 여러 클라이언트가 동시에 감지해도 한 명만 성공 (Postgres가 같은 행에 대한 동시 UPDATE를 직렬화해줌). 처음 계획했던 `turn_number` 가드는 더블/구매대기처럼 턴 안에서 세부 단계가 바뀔 땐 값이 그대로라 세밀도가 부족해서 `version`으로 대체함.
  4. 선점에 성공한 클라이언트만 실제 자동 행동(`ROLL_DICE` 또는 `DECIDE_PURCHASE{buy:false}`)을 제출.

  - **방장은 준비 토글이 없음** — 시작 버튼만 갖고, 시작 조건은 "방장을 제외한 전원 준비완료".
  - 로비 나가기 시 방장이 나가면 다음 순번에게 방장이 자동으로 넘어가고, 아무도 안 남으면 방이 삭제됨.

### 게임 (좌우 2열, GAME_UI_REDESIGN.md 기준 개편 완료)
중앙에 조작 카드를 띄우던 예전 방식은 폐기하고, **왼쪽 플레이 영역(보드+주사위+채팅) / 오른쪽 턴 패널** 2열 구조로 개편함.
- **플레이 영역**: 타원 트랙 보드가 중심. 보드 중앙에 CSS 픽 다이스(`DiceStage`) — 주사위 결과 + 현재 턴 플레이어 이름, 더블이면 "더블! 한 번 더". 보드 우측 상단 코너에 턴 번호/카운트다운(`TurnTimer`, 10초 이하 경고색). 보드 아래에 공용 채팅 입력창.
  - "로그 열기" 버튼은 아직 없음 — 채팅은 저장 안 하고 Broadcast로만 오가서 다시 볼 로그 자체가 없음 (`chat_messages` 테이블 항목 참고).
- **턴 패널**(`TurnPanel`): 보유 자산(현재 턴 플레이어 잔액, +/- 증감 표시) → 이벤트 메시지(notice, 종류별 색상 구분: 수입/지출/파산/황금열쇠) → 행동 버튼(단계별 주 행동+보조 행동, 패널 하단 고정) 순서. 참가자 전원 잔액 목록은 뺐음 — 말 배지로 이미 항상 보여서 중복이라 판단.
- 각 플레이어 말 위:
  - **금액 배지: 항상 표시** (말 색과 같은 테두리, `balance` 실시간 반영).
  - **채팅 말풍선: 채팅 시 잠깐(3.5초) 떴다 사라짐.** 길이 제한 20자. 테두리는 `border-color: inherit`로 부모(`.player-token`)의 인라인 색(자기 말 색)을 그대로 물려받음 — 누구 말풍선인지 테두리 색으로 구분됨.
    - **발견한 버그**: 말풍선이 "다음 칸" 타일에 가려 안 보이는 문제가 있었음 — `.player-token`이 인라인 `transform`(이동/스택 오프셋)을 갖고 있어서 새 스태킹 컨텍스트를 만드는데 정작 `z-index`가 없어서, DOM 순서상 나중에 그려지는 옆 `.board-cell`(타일)이 위에 덮여버렸음. `.player-token`에 `z-index:10`을 줘서 해결.
  - **말 위치**: 같은 칸에 혼자면 정중앙, 여럿이면 가로로 나란히(세로는 항상 중앙) 겹치지 않게 배치(`PlayerToken.tsx` 인라인 `transform: translate(calc(-50% + offset px), -50%)`). 예전엔 자기중심 정렬(-50%,-50%) 없이 스택 오프셋만 있어서 혼자 있어도 타일 중앙이 아니었던 버그가 있었음 — 발견 후 수정.
  - **이동 애니메이션**: 말이 새 위치로 순간이동하지 않고 한 칸씩 걸어서 이동(`src/hooks/useAnimatedPositions.ts`, 150ms/칸). 실제 게임 상태(`position`)는 그대로 두고 화면에 보여줄 위치만 별도로 서서히 목표를 따라가게 하는 순수 클라이언트 연출이라 동기화 로직과 무관. 더블 3연속→무인도 강제 이동은 원래 텔레포트라 애니메이션 없이 즉시 스냅, 황금열쇠 "뒤로 이동" 카드는 후진으로 걷고, 나머지는 전부 전진 — 서버가 이동 종류를 안 내려줘서 `notice` 문구로 구분(`TurnPanel.tsx`의 `classifyNotice`와 같은 패턴). 개발 모드 StrictMode의 effect 이중 실행으로 같은 말에 타이머가 두 개 생겨 두 배 속도로 움직이는 버그를 겪고 고쳤음(체크·등록을 setState 콜백이 아니라 ref로 동기 처리해서 해결) — Playwright로 50ms 간격 샘플링해 한 칸씩(5%p) 고르게 이동하는지 확인.
  - **주사위가 멈춘 뒤 걷게 하는 지연(hold)**: 주사위 애니메이션(`DiceStage`, `ROLL_MS`=850ms)이 도는 동안엔 말이 안 움직이고 주사위가 멈춘 뒤 걷기 시작하게 한다. **hold 판단은 반드시 `useAnimatedPositions` 훅 안에서 target 갱신과 같은 렌더/같은 effect로 처리해야 한다** — Windows에서 pull한 커밋이 이 hold를 `App.tsx`의 별도 `useEffect`(`moveHoldUntil` state)로 계산했는데, React가 자식(Board)의 애니메이션 effect를 부모(App)의 effect보다 먼저 실행하기 때문에 hold 값이 항상 한 렌더 늦게 도착 → 새 위치가 온 첫 렌더에선 hold가 직전 굴림 때의 과거 시각이라 `wait<0`이 되어 말이 즉시 걸어버리는 경쟁 상태가 있었다(증상: "주사위 애니메이션 진행 중에 말이 이동", 매 굴림마다 재현). App의 `moveHoldUntil` state/effect를 없애고, App이 `rollKey`만 계산해 Board→훅으로 내려주면 훅이 rollKey 변화를 target 갱신과 동시에 감지해 그때 hold를 건다.
  - **`rollKey`는 서버의 `rooms.roll_seq`(주사위를 실제로 굴릴 때만 +1 되는 카운터)를 쓴다** — 처음엔 `"d1-d2"`(주사위 눈)로 했는데 직전 굴림과 눈이 똑같으면(1/36) 값이 안 바뀌어 그 굴림만 hold가 안 걸리는 한계가 있었다. `roll_seq`는 눈과 무관하게 매 굴림 증가하므로 그 케이스까지 완전히 해결됨. 증가 지점은 `gameReducer`의 `ROLL_DICE` 케이스 한 곳(무인도/연속더블/일반 등 모든 굴림 갈래가 `handleRollDice`를 거치므로, 그 결과가 no-op(잘못된 phase)이 아닐 때만 `state.rollSeq + 1`). 구매/건축 등 굴림이 아닌 액션에선 증가하지 않는다.
  - **검증**: Playwright로 라이브(Edge Function 경유) 8회 굴림 — `roll_seq`가 1~8로 항상 증가하고, 같은 눈(2,5)이 두 번 나온 굴림에도 rolling 구간(주사위 애니메이션)이 첫 이동 시점보다 항상 먼저 끝나는 것(위반 0건) 확인. (게임 첫 굴림만은 `DiceStage`가 첫 lastRoll에선 애니메이션을 생략하는 기존 동작이라 주사위는 안 구르지만 hold는 정상 적용됨.)
  - 무인도 대기 중이면 "무인도 N"(남은 턴 수) 배지, 탈락자(파산 또는 자진 포기)는 흐리게 + "탈락" 표시.
- 좌우 코너(출발/무인도)는 아이콘 표시.
- 내 턴이 아니면 굴리기/구매 버튼 대신 "OO님 차례입니다" 문구만 보임 (다른 사람 턴에 조작 못 하게).
- 화면 하단에 "게임 포기하기" 버튼(탈락 전까지 항상 노출, 내 턴 여부 무관) — 확인창 한 번 거친 뒤 자진 탈락.

### 타일 카드 디자인 (구현 완료)
실제 부루마불 카드(상단 컬러/건물칸 + 흰 바탕 + 지명 + 국기 + 가격)를 레퍼런스로 보드 위 땅 칸(`empty_land`/`landmark`)의 생김새를 다시 짰다. 그 외 특수칸(출발/무인도/황금열쇠/사회복지기금/우주여행)은 라벨 하나만 중앙에 놓인 기존 방식 그대로(디자인 변경 범위 밖).

- **구조**(`Tile.tsx`): 상단칸(`.tile-top`, 건물 표시) + 흰 배경 본문(`.tile-body`, 지명 → 국기 → 가격/통행료 순). 본문 안 배치는 지명 위쪽 정렬 / 국기 가운데 정렬 / 가격·통행료 아래쪽 정렬 — `.tile-body`를 `justify-content: space-between`인 flex column으로 둬서 첫/마지막 요소가 각각 위·아래 여백 경계에 붙고 가운데 요소(국기)는 자연스럽게 중간에 남는다(처음엔 셋 다 `justify-content:center`로 뭉쳐 있었는데 피드백을 받고 이렇게 분리함).
- **상단칸 = 건물 자리표시자**: 지금은 실제 픽셀아트 에셋이 없어서 레벨별 이모지로 대신함(레벨1~2=🏠, 레벨3=🏢, 레벨4=🏨, 레벨0은 빈칸). 레이아웃 구조는 이미 완성돼 있어서, 나중에 진짜 건물 스프라이트가 생기면 이 자리 이미지만 갈아끼우면 됨. `landmark` 칸은 건물을 절대 못 지으므로 아이콘 대신 황금열쇠와 같은 금색 계열 배경으로 "특별한 땅"임을 표시(기존 배경 틴트를 상단칸으로 옮김).
- **국기**: 각 땅이 속한 나라의 국기. `src/game/board.ts`의 `TILE_FLAGS`(idx→ISO 3166-1 alpha-2 국가 코드)에 `LAND_DEFS`와 나란히 정의 — 게임 로직(엔진/리듀서)은 이 값을 전혀 참조하지 않는 순수 표시용이라 `Tile`(game/types.ts) 타입에는 안 넣고 별도 조회 테이블로 분리했다.
  - **처음엔 유니코드 국기 이모지를 썼다가 실제 SVG 파일(`flag-icons` npm 패키지, MIT)로 교체함** — Windows에서 국기 이모지가 그림 대신 빈 상자로 보이는 문제를 겪었음(플랫폼별 이모지 폰트 지원 차이, macOS는 Apple Color Emoji가 지원하지만 Windows Segoe UI Emoji는 지역표시문자 국기를 제대로 안 그려주는 경우가 있음).
  - `flag-icons`의 CSS(`fi`/`fi-{code}`)를 통째로 import하면 Vite가 CSS 안의 `url()` 참조를 전부(약 250개국 SVG) 번들에 넣어버리는 걸 실제로 겪어서, 실제 쓰는 26개국만 `src/game/flagAssets.ts`에서 개별 `import`해 `Record<code, url>`로 만들고 `Tile.tsx`가 인라인 `style={{backgroundImage: url("...")}}`로 직접 사용한다.
  - **발견한 버그**: 개발 모드에서 Vite가 작은 SVG를 `data:image/svg+xml,...` URI로 인라인하는데, 그 안에 홑따옴표(`'`)가 그대로 들어있어서 `url(...)`을 따옴표 없이 쓰면 CSS 파싱이 깨져 국기가 하나도 안 보였음(우연히 인라인 임계값을 넘겨 별도 파일로 나온 3개국만 보였음). `url("${...}")`처럼 쌍따옴표로 감싸서 해결.
- **가격/통행료 표시**: **주인이 없으면 땅값**(`tile.price`), **주인이 있으면(건물 유무·레벨 무관) 지금 통행료**를 보여준다 — 이미 팔린 땅의 "구매가"보다 "지금 내야 할 돈"이 더 유용한 정보라는 판단. 통행료 계산은 `src/game/buildings.ts::getCurrentToll(tileIdx, level)` 하나로 통일(기존엔 engine.ts 안에 인라인으로만 있던 걸 공용 함수로 뽑아서, engine.ts의 실제 징수와 Tile.tsx의 표시 둘 다 같은 계산을 쓰게 함 — TurnPanel.tsx 건설비 중복 구현 버그의 재발 방지와 같은 패턴).
- **크기 — 고정 px가 아니라 퍼센트+cqw 기반** (겹침 버그를 겪고 나서 확정한 설계):
  - 타일 폭은 `Board.tsx`가 `.board-cell`에 인라인으로 주는 `TILE_WIDTH_PCT`(`src/game/boardLayout.ts`, **4.4%** — 위/아래 줄 한 칸이 실제로 쓸 수 있는 폭(100%/20칸=5%) 중 88%를 채운 값, 카드를 좀 더 키워달라는 피드백으로 4.0→4.4 상향). `.tile`은 `width:100%`(=이 퍼센트) + `aspect-ratio:80/110`으로 높이를 비율로 따라가게 함.
  - **처음엔 타일을 고정 80×110px로 짰다가, 실제 Playwright 스크린샷 검증에서 40칸 중 36~40쌍이 겹치는 걸 발견**(game 모드 창 폭이 화면마다 다른데 타일 폭만 고정이면 안전 여백이 없어서). 그래서 폭을 퍼센트 기반으로 바꾸고 나서야 겹침이 0으로 확인됨 — 이 프로젝트에서 "고정 px 타일 + 가변 창 폭"의 조합이 실제로 깨지는 걸 확인한 첫 사례라 기록해둠.
  - `.board-cell`에 `container-type: inline-size`를 줘서, `.tile` 내부의 글자 크기(`.tile` font-size)·상단칸 아이콘 크기·국기 박스·글자·간격을 전부 `cqw`(컨테이너 폭 기준 단위)로 지정 — 타일이 화면마다 다른 실제 px 크기로 렌더돼도 내부 요소 비율은 항상 같다. 채팅을 오른쪽으로 옮겨 보드가 세로 공간을 더 쓸 수 있게 되고, "여전히 잘 안 보인다"는 피드백을 두 차례 받아 cqw 배율을 두 번 상향: 폰트 12.5→14→**16cqw**, 상단칸 아이콘 22.5→25→**27cqw**, 국기 박스 32.5×22.5→36×25→**40×28cqw**, 국기 글자 16.25→18→**20cqw**. 타일 폭(`TILE_WIDTH_PCT`)도 4.0→**4.4%**로 살짝 키움(여전히 슬롯의 88%만 채워 겹침 안전 여백 유지, 좌표로 재확인).
  - **코너(출발·무인도) 간격**: `boardLayout.ts`의 `CORNER_X_LEFT`/`CORNER_X_RIGHT`가 원래 3/97이라 코너(xPct 0/100)와 첫 줄 칸 사이 간격(7.7%)이 칸끼리 간격(4.7%)보다 훨씬 넓어서 "코너가 유독 멀어 보인다"는 피드백을 받았다. **0/100으로 바꿔서 코너도 같은 20칸 균등분할 격자에 포함**시켜 모든 간격을 통일함(검증: 코너-첫칸 간격과 칸-칸 간격이 정확히 일치하는 걸 좌표로 확인).
  - **보드 배경(알약 모양 테두리) 제거**: `.board`에 있던 `background`/`border`/`border-radius:999px`(타원 트랙 배경)를 걷어냈다 — "카드 뒤에 왜 타원이 있는지 모르겠다"는 피드백. 이제 타일 카드들이 배경 없이 떠 있는 모습.
- **채팅 위치**: 원래 왼쪽 플레이 영역(보드 아래)에 있던 채팅 입력창을 **오른쪽 열(턴패널 옆, `.right-column`)로 이동**(`App.tsx`) — 보드가 그만큼 세로 공간을 더 쓸 수 있게 돼서 타일이 커짐. `.turn-panel`은 이제 `.right-column`의 자식이라 자기 폭(320px) 대신 `flex:1`로 남는 세로 공간을 차지하고, 폭은 `.right-column`이 담당.
- 이 변경과 맞물려 **game 모드 Electron 창 폭도 화면 폭 전체를 쓰도록 바꿈**(아래 "Electron 패키징 > 화면별 동적 창 크기" 참고) — 높이는 그대로 디스플레이 1/3 고정.
- **검증**: `npm run dev` + headless Chromium(Playwright)으로 실제 방 만들어 게임 화면까지 진행 후 스크린샷 촬영, `getBoundingClientRect()`로 40칸 전부 겹치는 쌍이 있는지 좌표 기반 자동 확인(1366px/1920px 두 창 폭에서 0건 확인) + 코너 간격 좌표 확인 + 실제 이미지로 지명/국기/가격 표시 육안 확인.
- **범위 밖(다음 단계)**: 말/건물/아이콘을 실제 픽셀아트 에셋 + 정수 배율(1x/2x/3x)로 바꾸는 작업, 구역별 컬러(Zone 1~4) 표시.
- **미해결**: 사용자가 "로비 화면과 게임 화면의 높이가 안 맞는다"고 보고했는데, Playwright로 로비(440×360)와 게임(1366~1920×360) 양쪽의 `window.innerHeight`/`document.documentElement.scrollHeight`/`#root` 높이를 직접 측정해봐도 전부 정확히 일치해서(둘 다 360px) 재현하지 못했다. 실제 Electron 창에서만 나타나는 문제일 수 있어(이 도구 환경에선 Electron GUI를 못 띄움), 스크린샷이나 더 구체적인 설명이 필요.

### 반응형 크기 & 폰트 (구현 완료)
- Electron 위젯 창의 높이가 디스플레이 높이의 1/3로 고정되지 않고 모니터마다 달라지기 때문에(아래 "Electron 패키징" 참고), 화면 크기가 창 높이에 비례해서 커지고 작아지게 만들었다 — `--fluid-unit`(index.css) CSS 변수 하나로 닉네임/메인/로비/게임 화면의 카드 크기·글자 크기를 전부 통제.
  - 실사용 위젯 창 높이(보통 300~400px)에서는 `--fluid-unit`의 svh 비율 항보다 하한값(clamp 최솟값)이 거의 항상 이겨서, 하한 자체를 몇 차례 조정 끝에 `clamp(1.4rem, 3svh, 2rem)`로 잡았었다(처음 `clamp(0.5rem, 2.2svh, 1.1rem)`이 실제 창 높이에서 글자가 거의 안 보일 정도로 작았던 문제를 겪고 두 번 더 올림). 이 값은 **Dongle 전용 보정**이었음(Dongle이 같은 font-size에서도 다른 폰트보다 작아 보이는 경향) — Dongle을 Galmuri11 Bold + Noto Sans KR로 교체한 뒤 그 보정이 과해 전체적으로 커 보이는 문제가 생겨 `clamp(0.95rem, 2.4svh, 1.35rem)`로 다시 낮춤.
  - **예외**: 말 텍스트(`.player-token` 등)는 `--fluid-unit`을 안 쓰고 고정 px 유지 — 말/건물/아이콘은 나중에 픽셀아트 에셋으로 교체하면서 정수 배율(1x/2x/3x)로만 커지게 할 예정이라 지금은 손 안 댐. **타일(`.tile`)은 더 이상 이 예외가 아님** — 타일 카드 디자인을 새로 짜면서 고정 px를 버리고 퍼센트+cqw 기반으로 바꿨다(아래 "타일 카드 디자인" 참고).
- **폰트**: 처음엔 Dongle 단일 폰트를 썼으나, 이후 **Galmuri11 Bold + Noto Sans KR 이원화로 교체**(Dongle 파일은 삭제). 둘 다 OFL 라이선스, CDN 대신 `src/assets/fonts/`에 직접 포함시켜 오프라인에서도 항상 같은 폰트로 보이게 함(`index.css`의 `@font-face`, `NotoSansKR-{Regular,SemiBold,Bold}.ttf` + `Galmuri11-Bold.woff2`).
  - **역할 분담**: `--font-emphasis`(Galmuri11 Bold, 픽셀 서체) — 타일 이름(`.tile-label`)/현재 턴(`.dice-stage-player`, `.turn-timer-number`)/구매·건축·주사위 버튼(`.turn-panel-actions button`)/황금열쇠 메시지(`.turn-panel-notice--gold`)/게임 결과 제목(`.result-screen h1`)에만 적용. 그 외 전부(채팅/긴 이벤트 설명/설정/부가 설명 등)는 `--sans`(Noto Sans KR)가 기본값으로 적용됨.
  - `font-synthesis: none`이 설정돼 있어(가짜 볼드 금지) Noto Sans KR은 실제 코드에서 쓰는 굵기(400/600/700) 파일을 전부 `@font-face`로 등록해둬야 의도한 굵기로 보인다 — 하나라도 빠지면 그 굵기 요청이 다른 등록된 굵기로 대체되어 보임.
  - Noto Sans KR ttf 3개 굵기 용량이 꽤 크다(굵기당 약 6.2MB, 합쳐서 약 18.5MB) — Dongle 대비 패키징 용량이 늘었음. 지금은 기능 우선으로 그대로 두지만, 배포 용량이 신경 쓰이면 나중에 woff2로 바꾸거나 서브셋(실제 쓰는 글자만 추출)하는 걸 고려할 것.
  - 라이선스 텍스트 파일(OFL.txt)은 아직 안 넣음 — Dongle 때는 있었는데 이번 교체 때 누락, 배포 전에 챙길 것.
- 서버-클라이언트 시계 차이 보정: 여러 PC로 테스트하면서 로컬 시스템 시계가 서로 어긋나 있으면 턴 타이머 표시(몇 초 남았는지)가 PC마다 다르게 보이는 문제를 발견 — `src/lib/serverClock.ts`의 `getServerNow()`가 보정된 "지금"을 계산해서 반환한다. `src/lib/supabaseClient.ts`가 `createClient`에 커스텀 `fetch`(`fetchAndSyncClock`)를 넘겨서, Supabase로 나가는 **모든** 요청의 응답을 가로챌 때마다 `updateClockOffsetFromResponse`로 서버 시계 오차를 계속 갱신한다 — 그래서 앱을 켜두는 동안 오차 추정치가 계속 최신화됨. `TurnTimer`(화면 표시)와 `useSyncedGame`의 타임아웃 감지 주기(위 "자동 진행" 참고) 둘 다 이 값을 쓴다. 실제 자동진행 최종 판정은 서버(Edge Function)가 자기 시계로만 하므로 이 보정과 무관하게 정확하다.

> 목업은 방향 확인용일 뿐. 색/간격/말풍선 스타일 등 세부는 구현하며 자유롭게 다듬어도 됨.

## Electron 패키징 (데스크톱 위젯, 4단계 P2 — 상당 부분 구현 완료)

원래 로드맵은 3단계를 다 끝내고 4단계를 시작하는 순서였지만, 3단계 마지막 항목(독점 보너스)이 플레이 판단 대기 상태라 Electron 작업을 먼저 진행했다. "다른 PC에 실행 파일을 보내서 같이 플레이"가 목표였고, 트레이 아이콘·내 턴 알림·캐릭터 커스텀은 다음 슬라이스로 남겨뒀다.

### 창 형태
- `electron/main.cjs`가 메인 프로세스. `BrowserWindow`를 `frame:false`(테두리 없음) + `transparent:true`(배경 투명) + `alwaysOnTop:true` + `resizable:false`로 띄운다.
- 위치는 항상 **주 모니터 작업 영역(작업표시줄 제외)의 우측 하단**(`screen.getPrimaryDisplay().workArea` 기준 계산, 여백 0 — 화면 끝에 딱 붙음).
- 높이는 항상 **디스플레이 높이의 1/3**(`HEIGHT_RATIO = 1/3`)로 고정 — 세로는 절대 안 늘림. 폭은 화면에 따라 동적으로 바뀐다(아래 참고).
- 투명 창이 제대로 보이려면 웹페이지 자체도 배경을 안 칠해야 해서(`index.css`의 `html`/`body`/`#root` `background: transparent`), 원래 전체 화면에 깔던 아이보리색 배경을 지웠다. 실제 카드(`.nickname-screen` 등)는 자기 배경색이 있어서 그대로 보이고, 빈 여백만 데스크톱이 비친다.

### 화면별 동적 창 크기 (구현 완료)
닉네임/메인/로비 화면은 카드 폭(440px, `COMPACT_WIDTH`)만 있으면 되는데, 게임 화면은 타일 카드 디자인이 커지면서(아래 "타일 카드 디자인" 참고) 보드 자체가 훨씬 넓은 폭을 필요로 한다.
- **game 모드 폭 = 화면 폭(`workArea.width`) 그대로** (구현 완료, `electron/main.cjs`). 예전엔 1360px 고정값이었는데, 타일이 훨씬 커지면서(고정 38px → 화면 폭에 비례하는 %) 1360px로는 40칸이 겹쳐버려서 창 폭 자체를 화면이 허용하는 만큼 쓰도록 바꿨다. 세로(높이)는 여전히 디스플레이 높이의 1/3로 고정 — 가로만 늘어나고 세로는 안 늘어난다는 원칙.
- `electron/preload.cjs`: `contextBridge`로 `window.electronAPI.resizeWindow(mode)` 하나만 렌더러에 안전하게 노출(`contextIsolation:true` 유지, Node 접근은 그대로 차단).
- `src/App.tsx`: `screen` state(로딩/닉네임/메인/로비/게임)가 바뀔 때마다 `useEffect`로 `resizeWindow('game' | 'compact')` 호출. 이 API는 Electron에서만 존재해서 일반 브라우저 탭에서는 그냥 무시된다(옵셔널 체이닝).
- 메인 프로세스는 `ipcMain.on('resize-window', ...)`로 받아서 `computeBounds(mode)`로 폭·높이·위치를 다시 계산해 `setBounds()` — 우측 하단 고정을 유지한 채로 크기만 바뀐다. `resizable:false`는 사용자가 직접 드래그로 리사이즈하는 것만 막을 뿐, 프로그램이 `setBounds`를 호출하는 건 막지 않는다.
- `.game-layout`(App.css)도 예전 `min(1400px, 98vw)` 폭 상한(창이 1360px 고정이던 시절 값)을 걷어내고 `width:100%`로 창을 그대로 채우게 바꿨다.

### 드래그 & 닫기 (프레임이 없어서 직접 구현, 구현 완료)
- `#root` 전체를 `-webkit-app-region: drag`로 두고, 실제 화면 카드(`.nickname-screen`/`.main-screen`/`.lobby-screen`/`.game-screen`/`.loading-screen`)만 `no-drag`로 다시 뚫었다. 카드 바깥의 빈 여백(투명하게 비치는 부분)을 아무 데나 잡으면 창이 움직인다 — 별도 드래그 손잡이 UI 없이 해결.
- `src/components/WindowChrome.tsx`: 프레임이 없으면 닫기 버튼 자체가 없어지므로 우측 상단에 작은 × 버튼을 직접 띄움(`window.close()` 호출 — Electron 렌더러에서는 이 표준 DOM API가 실제 창 닫기로 이어짐, IPC 불필요).

### 빌드 & 배포 (Windows 포터블 zip, 구현 완료)
- `vite.config.ts`: `base: './'` 필수 — Electron 프로덕션은 `file://`로 `dist/index.html`을 여는데, 기본값(절대경로 `/assets/...`)이면 애셋을 못 찾고 흰 화면만 뜬다. `server.strictPort: true`로 개발 서버 포트(5173)도 고정.
- `npm run electron:build` = `vite build`(프로덕션 웹 번들) → `electron-builder --win zip --x64`.
  - **`--x64`를 반드시 명시할 것** — 이 프로젝트 개발 환경(Apple Silicon Mac)은 arm64라, 생략하면 host 아키텍처 그대로 arm64 Windows용으로 잘못 빌드된다.
  - 타깃은 `zip`(포터블) — 설치 마법사 없이 압축 풀고 exe 더블클릭. wine 설치 없이도 macOS에서 빌드된다(NSIS 설치파일 방식은 wine이 필요해서 채택 안 함).
  - `package.json`의 `build.files`에서 `!node_modules/**/*`로 제외 — 메인 프로세스(`main.cjs`/`preload.cjs`)가 npm 패키지를 전혀 `require`하지 않아서(렌더러 코드는 이미 Vite가 `dist/`에 다 번들링해둠) node_modules를 통째로 넣을 필요가 없다.
  - 결과물은 `release/`(gitignore 처리, 커밋 안 됨) 아래 `win-unpacked/`와 `*.zip`.
  - **코드 서명은 안 함(확정)** — 유료 인증서가 필요해서 개인 배포 범위 밖. 상대방이 실행할 때 Windows SmartScreen 경고("추가 정보 → 실행")나 Defender 오탐이 뜰 수 있는데 정상이라고 미리 안내할 것.
  - 자동 업데이트(electron-updater 등)도 아직 안 함 — 지금은 새 zip을 다시 보내는 걸로 충분.
- 개발: `npm run electron:dev` = `concurrently`로 `vite`(개발 서버)와 `wait-on http://localhost:5173 && electron .`을 같이 실행. `wait-on`이 없으면 Vite가 뜨기 전에 Electron이 먼저 열려서 흰 화면이 뜨는 경쟁 상태가 생긴다.

### Claude Code 세션의 한계 (다음 세션도 참고)
Claude Code가 실행되는 이 도구 환경은 진짜 GUI 프로세스를 못 띄운다(`require('electron')`이 정상 앱 객체 대신 문자열을 반환 — Electron이 진짜 Electron 실행파일이 아니라 plain Node로 실행됐을 때 나타나는 문서화된 현상). 샌드박스를 꺼도 `codesign_util.cc` 관련 에러가 나며 안 됨. 그래서:
- `npm run electron:dev`로 실제 창이 잘 뜨는지, 투명·드래그·크기 전환이 제대로 되는지는 **항상 사용자가 직접 눈으로 확인**해야 한다.
- `npm run electron:build`(패키징)까지는 검증 가능하다 — `file` 명령으로 실제 Windows x64 실행파일인지, `unzip -l`/`npx asar list`로 산출물 구조가 맞는지는 도구에서 직접 확인 가능.

## 데이터베이스 스키마 (2단계에서 Supabase에 생성)

> 성격 구분: **신원**(사람) / **게임상태**(이번 판 한정) / **외형**(표시용 꾸미기).

### `rooms`
`id` uuid PK · `code` text · `status` text(waiting/playing/finished) · `turn_time_sec` int · `current_player_id` uuid · `turn_number` int · `turn_started_at` timestamp · `host_client_id` text · `created_at` timestamp

2단계 상태 동기화/자동진행을 위해 추가된 컬럼(엔진의 `GameState`를 그대로 담기 위함, `src/game-sync/mapping.ts` 참고):

`awaiting-build-decision`(재방문, 한 번에 한 단계 Y/N) / `awaiting-initial-build-decision`(최초구매 직후, 감당되는 등급 중 목표를 한 번에 선택 — 액션 `DECIDE_INITIAL_BUILD{targetLevel}`) / `awaiting-start-bonus-build`(출발칸 정확 도착, 보유 땅 중 하나 선택)는 3단계 건물 시스템에서 추가됨 — 새 컬럼 없이 기존 컬럼 재사용.

`awaiting-space-travel-destination`(우주여행 칸 도착 → 목적지 선택, 액션 `DECIDE_SPACE_TRAVEL{tileIdx}`)도 새 컬럼 없이 기존 컬럼만으로 구현 완료 — 대상 칸을 그때그때 고르는 방식이라 출발칸 보너스와 마찬가지로 `pending_purchase_tile_idx`를 안 쓰고 액션 자체에 `tileIdx`를 실어 보냄.

`event_deck` jsonb (**구현 완료** — 황금열쇠 덱의 남은 카드 순서. 셔플 결과를 서버가 들고 있어야 클라이언트가 다음 카드를 미리 못 봄. 3단계에서 처음으로 기존 컬럼 재사용이 아니라 진짜 새 컬럼을 추가한 사례 — 사용자가 SQL Editor에서 직접 마이그레이션 실행) · `welfare_pool` int (**구현 완료** — 사회복지기금 적립액, 같은 방식으로 마이그레이션) · `consecutive_doubles` int (**구현 완료** — 더블 3연속 → 무인도 판정용, 턴이 실제로 다음 사람에게 넘어갈 때 0으로 리셋, 같은 방식으로 마이그레이션) · `roll_seq` int (**구현 완료** — 주사위를 실제로 굴릴 때만 +1 되는 카운터. 클라이언트가 "새 굴림"을 정확히 식별해 말 이동 애니메이션을 주사위가 멈춘 뒤로 미루는 데 씀 — 위 "타일 카드 디자인 > 이동 애니메이션" 참고. 같은 방식으로 마이그레이션).

### `tiles` (40칸 보드 정의, 고정 마스터 데이터)

## 보안 (2단계에서 완료)

- **규칙 계산은 이제 전부 `supabase/functions/game-action`(Deno Edge Function, 서비스 롤 키)이 담당한다.** `ROLL_DICE` / `DECIDE_PURCHASE` / `CLAIM_TIMEOUT`(턴 타이머 자동진행) / `FORFEIT`(포기하기) 네 가지 액션 전부 이 함수 하나로 들어옴. 함수 안에서 `src/game/engine.ts`의 `gameReducer`를 그대로 실행(코드 중복 없이, 상대경로 import로 재사용 — Deno라 내부 import에 `.ts` 확장자가 필요해서 `engine.ts`/`board.ts`/`config.ts`/`types.ts`/`game-sync/mapping.ts`/`lobby/types.ts`만 그렇게 고쳐둠)한 뒤, 결과를 `rooms.version` 낙관적 락과 함께 DB에 반영한다.
  - 요청마다 서버가 직접 검증: `START_GAME`은 요청한 client_id가 `host_client_id`와 같은지, `ROLL_DICE`/`DECIDE_PURCHASE`는 요청한 client_id가 지금 턴인 플레이어와 같은지. 클라이언트가 브라우저 콘솔에서 이 검증을 우회할 방법이 없음(단, client_id 자체를 위조하는 것까진 못 막음 — 로그인이 없어서 생기는 한계, 아래 참고).
- **RLS + 컬럼 단위 권한으로 실제로 잠금.** `rooms`/`players`/`ownerships`에 RLS 활성화:
  - 읽기(select)는 anon 전부 허용 (로비/게임 화면 렌더링 + Postgres Changes 구독용).
  - `ownerships`는 anon용 insert/update/delete 정책이 아예 없음 — Edge Function(서비스 롤, RLS 항상 우회)만 쓸 수 있음.
  - `rooms`/`players`는 로비 CRUD(방 생성/참가/나가기)를 위해 insert/delete/update 정책은 있지만, **실제로 어느 컬럼을 고칠 수 있는지는 RLS가 아니라 컬럼 단위 `GRANT`로 잠갔다**: `rooms`는 `turn_time_sec`/`host_client_id`만, `players`는 `is_ready`만 anon이 직접 수정 가능. `balance`/`position`/`phase`/`version` 같은 게임 상태 컬럼은 anon 키로는 UPDATE 자체가 거부됨(Edge Function만 가능).
- **한계**: 로그인이 없어서 client_id 위조까진 못 막는다 — "내가 다른 사람인 척 요청을 보낸다"는 시나리오는 여전히 가능. 이건 CLAUDE.md가 애초에 말한 "부정행위(계산 조작)"보다 더 깊은 문제라 로그인 도입 전까진 범위 밖으로 남겨둠.
- 개발/배포: Supabase CLI 설치·로그인·프로젝트 링크 완료(`supabase link --project-ref qtfzhpafpsbnhjpkwdjj`). 배포는 Docker 없이 `supabase functions deploy game-action --use-api`로 잘 됨(이 프로젝트 규모에선 Docker 불필요했음).

## 미정 사항 (구현하며 결정)

- 밸런스 수치 — **원작 환산 기준으로 전면 재설계 + 코드 반영 완료**. 환산: 원작 1만 원 = 우리 10.
  - **초기자본 1500 → 3000**, 월급 200 유지. 근거: 원작 초기자본 243~293만 원 / 월급 20만 원 = **12~15배**인데 우리는 7.5배였다. 월급이 상대적으로 두 배 후해서 파산이 안 났던 것.
  - **땅값**: 등차수열 100~700 → 구역별 계단식 60~350(건물 가능 25칸) + `landmark` 6칸(200~900).
  - **건설비**: 땅값 비례 → 구역별 고정 절대값.
  - **통행료 배율**: 빌딩 ×3→×4, 호텔 ×4.5→×8.
  - 검산: 월급 ÷ 최고 건물땅 = 200/350 = 0.57배로 원작(20/35 = 0.57배)과 정확히 일치. 4인 초기자본 총합 대비 전체 땅값 비율도 66%로 원작 63%와 근사.
  - **재시뮬레이션 완료** (2인전/4인전 각 50판, 자동 플레이 스크립트로 측정 — 1단계 때 만든 시뮬레이션 코드는 실제로 존재하지 않아 이번에 새로 작성, 매 실행 후 삭제하는 임시 스크립트 패턴 그대로 유지).
    - **봇 정책(중요, 결과 해석의 전제)**: 방치형의 "타임아웃 자동 기본값(아무것도 안 함)"이 아니라 **"살 수 있으면 항상 사고, 지을 수 있으면 항상 최대 등급까지 짓는" 적극적 투자 정책**으로 굴렸다. 자동 기본값 그대로 굴리면 아무도 안 사고 안 지어서 경제가 전혀 안 돌아가 무의미한 결과가 나오기 때문 — 즉 이 수치는 "다들 활발하게 플레이할 때"의 밸런스를 보여준다.
    - **2인전(50판)**: 평균 게임 길이 약 65~69턴(최소 12 / 최대 166, 판마다 편차 큼) · 평균 최종 승자 자산 약 420~430. 파산 발생률은 승리 조건이 "최후 1인 생존"뿐이라(턴 제한 옵션 미구현) 정상 종료된 게임은 정의상 100% 파산으로 끝난다 — 그래서 의미 있는 지표는 "몇 턴에 파산이 나는가"이고, 그게 곧 평균 게임 길이와 같다.
    - **4인전(50판)**: 평균 게임 길이 약 122~123턴(최소 55 / 최대 292) · 평균 최초 파산 발생 턴 약 38~44턴(첫 탈락자가 나오는 시점, 게임 자체는 3명이 전부 파산할 때까지 계속됨) · 평균 최종 승자 자산 약 1590~1810.
    - **정상 종료율**: 두 구성 모두 100/100판이 4000액션 안전장치에 걸리지 않고 정상적으로(파산으로 승자가 결정되어) 끝남 — "파산이 하나도 안 나서 게임이 안 끝나는" 옛 1단계 문제는 이번 구조에서는 재현되지 않았다.
